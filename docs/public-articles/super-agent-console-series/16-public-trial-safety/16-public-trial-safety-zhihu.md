标题

把 Agent 发给朋友试用前，我先给它加了几道保险

正文

真实模型接进来之后，我开始有点想把这个项目发给朋友试用。

毕竟自己在本地跑，和别人真的打开网页使用，是两种完全不同的状态。

自己用的时候，输入什么、点几次、什么时候停，都是可控的。

但一旦发到公网，就不一样了。

别人可能连续点很多次。

也可能输入一大段内容。

更麻烦的是，如果链接被转发出去，我根本不知道会有多少人访问。

而这个项目背后接的是真实大模型。

每一次请求都可能消耗 token。

服务器本身也只是轻量应用服务器，不是什么高配机器。

所以在真正发出去之前，我先停下来想了一下：

这个网站至少要有哪些保护，才不至于一公开就失控？

## 第一道保护：访问码

我没有做完整的注册登录系统。

因为这个阶段只是朋友内测，没有必要上手机号、验证码、用户体系这些东西。

我采用的是 Access Code。

朋友打开网站后，需要输入一个我提前准备好的访问码。

访问码正确，服务端会写入一个 `httpOnly cookie`。

之后访问页面和接口，就靠这个 cookie 判断身份。

这里我还加了一个可选昵称。

朋友登录的时候，可以顺手填一个昵称。

昵称不是凭证，不需要加密。

它只是为了后面提交反馈时，我知道是哪位朋友提的意见。

这一步很小，但对内测很有用。

因为如果只看到一堆匿名反馈，其实很难追问上下文。

比如有人反馈：

```txt
刚刚点了之后没反应
```

如果我知道是谁提的，就可以继续问：

```txt
你当时输入的是什么？
页面停在哪一步？
有没有刷新？
```

这比完全匿名反馈要好很多。

## 第二道保护：普通用户和管理员区分

普通朋友只需要使用 Agent。

他们不应该看到所有人的会话记录。

但我自己在线上又需要排查问题。

我需要看：

```txt
Run 列表
Conversation 列表
Feedback 列表
Run 详情
```

所以我增加了管理员访问码。

普通访问码登录后，只能使用主功能和提交反馈。

管理员访问码登录后，才能看到管理页面。

这一步让我意识到：

即使是一个个人 Demo，也不能完全没有权限边界。

如果公开出去之后，任何人都能看到所有测试记录，那肯定是不合适的。

以前我做本地 Demo 时，很容易忽略这个问题。

因为自己用的时候，所有数据都是自己的。

但只要有第二个人进来，数据边界就出现了。

哪怕只是朋友内测，也应该把普通用户和管理员分开。

## 第三道保护：频控

接下来是频控。

这部分我加了几层：

```txt
单用户每分钟最多创建多少个 Run
单用户每天最多创建多少个 Run
全站每分钟最多创建多少个 Run
全站每天最多创建多少个 Run
```

这样即使某个访问码被多人共用，或者有人连续点击发送，也不至于无限消耗模型 token。

这里有一个细节：

如果多个人使用同一个 access code，他们在系统里会被识别成同一个用户。

也就是说，他们共享同一个 `userId`，也共享同一套频控额度。

这对内测阶段反而是合理的。

因为我可以给不同朋友发不同 code。

如果某个 code 的使用量异常，也比较容易定位。

这不是一个完整用户系统。

但对当前阶段来说够用了。

我的目标不是做账号体系。

我的目标是让公网试用不要失控。

## 第四道保护：并发限制

频控限制的是“创建次数”。

但 Agent Run 一旦开始，SSE 会持续连接，模型也可能持续生成。

如果很多 Run 同时执行，服务器和模型请求都会有压力。

所以我又加了并发限制：

```txt
单用户同时最多几个 Run
全站同时最多几个 Run
```

这样至少能避免几个长任务同时把服务器拖住。

这一步和频控不一样。

频控解决的是：

```txt
一段时间内调用太多
```

并发限制解决的是：

```txt
同一时间正在执行的任务太多
```

这两个都需要。

因为 Agent 不是普通短接口。

它可能会持续输出。

它可能会调用模型。

它可能会执行 Tool 和 Skill。

如果不限制并发，一旦多个任务同时跑起来，轻量服务器很容易扛不住。

## 第五道保护：输入长度限制

大模型接口最怕的一种情况，就是用户一口气贴非常长的内容。

比如一本书。

几十页文档。

一大段日志。

或者直接把一整篇文章复制进去。

这会直接把 token 消耗打上去。

所以我加了：

```txt
MAX_RUN_INPUT_LENGTH
```

超过长度就不创建 Run。

这不是完美方案。

比如更合理的做法可能是：

```txt
上传文件
文档解析
分片
RAG
摘要
```

但那是后面的能力。

内测阶段，我先用最简单的方式保护住模型调用成本。

不要让一个超长输入直接把 token 打爆。

这一步很实用。

## 第六道保护：模型防灾开关

这个我觉得非常重要。

我加了一个环境变量：

```txt
MODEL_ENABLED=false
```

一旦关掉，网站还可以打开。

登录还能用。

反馈也还能用。

但用户创建 Agent Run 时，会直接返回：

```txt
模型能力暂时关闭
```

这样如果我发现：

```txt
token 消耗异常
模型服务异常
服务器压力异常
访问码泄露
公网请求突然变多
```

可以先一键止血。

不需要立刻删代码。

也不需要停整个网站。

这一步做完，我心里踏实很多。

因为真正公开一个接大模型的项目时，最怕的不是功能有 bug。

而是出问题时没有开关。

没有开关，就只能临时改代码、重新部署，或者直接停服务。

而有了这个开关，至少可以先把成本风险控制住。

## 最后补了反馈功能

除了这些保护，我还加了一个反馈功能。

页面右下角有一个反馈入口。

朋友试用完，可以直接提交建议、问题或者感受。

这些反馈会落到数据库里。

管理员可以在后台页面查看。

这个功能本身不复杂。

但它让我感觉这个项目开始从“自己玩”变成了“可以让别人参与”。

因为别人不是只能用完之后微信告诉我一句：

```txt
还行
```

而是可以在使用现场，把问题直接留下来。

这对内测很重要。

因为问题通常发生在现场。

如果离开现场再回忆，很多细节就没了。

比如：

```txt
点了哪个按钮
输入了什么
页面停在哪一步
是否有返回
是否刷新过
```

反馈入口虽然简单，但它让问题更容易被收集下来。

## 做完这些，我才敢把链接发出去

这一阶段做完之后，我才比较敢把链接发出去。

因为我知道，即使有人连续点、输入很长内容，或者链接被转发出去，项目也不至于完全失控。

至少有几道保护：

```txt
访问码
管理员权限
用户频控
全站频控
用户并发限制
全站并发限制
输入长度限制
模型防灾开关
反馈入口
```

这些东西听起来不像核心功能。

它们没有 Tool Calling 那么像 Agent。

也没有 SSE Timeline 那么直观。

更没有真实模型接入那么让人兴奋。

但它们决定了一个项目能不能从本地走到公网。

## 很多 Demo 不是死在做不出来，而是死在一公开就失控

这一阶段给我的感受是：

很多个人项目不是死在功能做不出来。

而是死在一公开就失控。

比如：

```txt
接口被频繁调用
模型 token 被刷掉
服务器被并发拖死
用户看到不该看的数据
线上出问题没有开关
反馈收不到
```

这些问题如果不提前想，等真的发生时会很被动。

以前我做项目，很多时候只关心功能有没有完成。

页面能不能打开。

接口能不能返回。

流程能不能跑通。

但这次我开始更关注另一个问题：

```txt
当它真的被别人访问时，会不会出事？
```

这可能也是从“写一个 Demo”到“做一个可试用产品”之间，必须补上的一课。

本地能跑，只能说明功能通了。

公网可试用，才开始考验边界。

这一步让我更清楚地意识到：

一个 AI Agent 产品，不能只考虑它能不能回答。

还要考虑：

```txt
谁能访问
访问多少次
能不能并发
输入有没有限制
成本会不会失控
出问题能不能关闭
反馈能不能收集
数据边界是否清楚
```

这些看起来是外围能力。

但它们其实是产品真正能被别人使用前，必须补上的保险。
