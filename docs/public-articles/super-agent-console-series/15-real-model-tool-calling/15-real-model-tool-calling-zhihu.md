知乎版

标题
我第一次把真实大模型接进 Agent，才发现不只是调一个接口
正文
前面几天，我已经把 Agent 的基础链路跑起来了。
从页面输入一段内容，到服务端创建 conversation、message、run，再通过 SSE 把 Agent 执行过程实时推回前端，这条链路基本已经成型。
同时，我也把 Agent 的每一步都落进了 MySQL。
一次 Run 里发生了什么，什么时候开始，什么时候调用模型，什么时候调用工具，Skill 执行了什么，最终结果是什么，都可以在数据库里查到。
到这一步，看起来已经有点像一个真正的 Agent 项目了。
但其实还有一个很关键的问题：
前面跑通的，大部分还是 Mock。
也就是说：
流程是真的
架构是真的
事件是真的
数据库也是真的
但 Agent 的“大脑”还不是真的
所以接下来，我开始把真实大模型接进来。
一开始我以为这件事很简单。
不就是调一个大模型接口吗？
准备好 API Key，把用户输入传过去，拿到返回结果，再展示到页面上。
但真正开始做的时候，我才发现，Agent 项目里的模型调用，和普通聊天接口并不是一回事。
普通聊天接口，大概是这样：
用户输入
↓
请求模型
↓
返回答案
而我现在要做的 Agent 链路，更像是这样：
用户输入
↓
模型判断是否需要调用工具
↓
服务端校验工具和参数
↓
执行 Tool
↓
Tool 内部编排 Skill
↓
Skill 可能继续调用模型
↓
Tool Result 回填给模型
↓
模型生成最终答案
它不是一次请求。
而是一条执行链。
先做 Model Adapter
我这次接入的是火山方舟上的豆包 Seed 2.0 Lite。
如果只是为了尽快跑通，最直接的做法是：
把火山方舟的接口请求直接写进业务代码里。
但我没有这么做。
原因很简单：
现在接的是豆包，但以后可能会换成 DeepSeek、通义千问、智谱，甚至同一个模型也可能通过不同平台接入。
如果我在 Agent Runtime 里直接写死某个平台的 HTTP 请求，后面一换模型，整个业务逻辑都会被污染。
所以我先做了一层 Model Adapter。
我把模型分成两层理解：
模型平台：火山方舟、阿里百炼、腾讯混元、OpenAI 等
具体模型：豆包 Seed、DeepSeek、通义千问、智谱 GLM 等
项目里对应的配置变成了：
MODEL_PROVIDER=volcengine_ark
MODEL_NAME=doubao-seed-2-0-lite-260428
MODEL_API_KEY=xxx
MODEL_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
这样 Agent Runtime 只关心一件事：
我要调用一个模型适配器
至于这个适配器后面是豆包，还是别的模型，那是配置和 Adapter 的事情。
这一步看起来只是多封装了一层。
但对 Agent 项目来说，这层很重要。
因为模型接入本身就应该是可替换的。
Agent Runtime 不应该被某一个模型平台绑死。
大模型不只是生成答案
接着，我开始处理 tool call。
这一步让我对 Agent 的理解又往前走了一点。
以前我理解的大模型，是“生成答案”。
但在 Agent 里，大模型还要参与“决策”。
比如用户输入一段岗位 JD 后，模型需要先判断：
这是不是一个适合调用工具的任务？
如果是，它应该返回类似这样的信息：
我要调用 analyzeJobAndGeneratePlan 这个工具
参数是 jdText 和 candidateProfile
服务端收到之后，不能直接信任模型。
因为模型可能返回不存在的工具名。
也可能返回不符合 schema 的参数。
所以服务端要做三件事：
检查 toolName 是否在白名单
校验 arguments 是否符合 Tool Schema
把请求路由到对应 Tool Handler
这就是 Tool Router 的作用。
以前我觉得 tool call 好像就是模型返回一个函数名，然后服务端执行。
但真正写的时候才发现，中间必须有一层边界。
模型可以提出调用意图。
但真正能不能调用、参数是否合法、应该路由到哪个处理器，必须由服务端控制。
否则模型一旦返回错误工具名或错误参数，后面的执行链路就会失控。
Tool 内部继续执行 Skill
Tool 被路由之后，内部还要继续执行 Skill。
目前我做了两个 Skill：
extractJobRequirementsSkill
generateSevenDayPlanSkill
第一个负责拆岗位要求，比如：
必备技能
加分项
岗位职责
风险点
第二个负责根据岗位要求和候选人背景生成准备计划，比如：
第 1 天做什么
第 2 天做什么
第 3 天做什么
一开始，这两个 Skill 是固定逻辑。
也就是我在代码里写死了一些模拟结果。
这样做适合跑通流程，但它毕竟还是 Mock。
后来我把它们改成了 model 模式。
也就是说，Skill 自己内部也会调用模型，让模型生成结构化 JSON。
这样整个链路就变成了：
模型决定调用 Tool
↓
Tool 编排 Skill
↓
Skill 调模型生成结构化结果
↓
Tool 汇总结果
↓
模型基于 Tool Result 生成最终答案
这一步跑通之后，项目终于不再只是一个 Mock Agent。
它开始真的有点像一个轻量版智能体运行时了。
怎么知道模型有没有真的调用工具？
真实模型接进来之后，第一个问题很快出现了。
怎么判断模型有没有真的返回 tool call？
一开始我只看最终回答。
页面上确实有答案，感觉模型好像工作了。
但我并不知道它中间有没有走工具。
也不知道模型有没有做工具规划。
如果只看最终答案，很多中间过程都是黑盒。
所以后来我在 Timeline 里加了更明确的事件：
model_call_start
model_tool_call_decision
tool_call_start
skill_start
skill_result
tool_call_result
final_answer_delta
这样我就能看到模型到底有没有做工具规划。
如果模型没有返回 tool call，我能看到。
如果 Tool Router 拒绝了非法工具，我也能看到。
如果 Skill 执行失败，也能定位到具体是哪一步。
这也是 Timeline 的意义。
它不是为了让页面看起来更热闹。
而是为了让 Agent 的执行过程可观察。
对 Agent 来说，只看最终答案是不够的。
因为最终答案只是结果。
真正决定这个结果质量的，是中间那条执行链。
JSON 输出也不是完全稳定
第二个问题是 JSON 输出不稳定。
Skill 要求模型返回 JSON。
理想情况下，它应该返回类似这样的结构：
{
"requiredSkills": ["Vue", "Node.js", "SSE"],
"bonusSkills": ["Agent", "RAG"]
}
但真实模型偶尔会包一层 Markdown，或者在 JSON 前后加一些解释。
比如它可能返回：
下面是分析结果：

````json
{
  "requiredSkills": ["Vue", "Node.js", "SSE"],
  "bonusSkills": ["Agent", "RAG"]
}

对于人来说，这没什么问题。

但对于服务端来说，如果直接 `JSON.parse`，就会失败。

所以我做了一个简单解析兜底：

```txt
优先解析 JSON code block
如果没有，就从文本里提取第一个 JSON 对象
如果仍然失败，就记录错误事件
这不是最终生产级方案。
但对于 MVP 来说够用了。
它至少能让我继续往前验证完整链路。
这件事也让我意识到：
让模型输出结构化数据，不是写一句“请返回 JSON”就结束了。
服务端必须考虑解析、兜底、错误记录和后续可恢复。
流式事件太多，也会变成问题
第三个问题是流式事件太多。
真实模型是一个 token 一个 token 往外吐。
如果每个 token 都落成一个 AgentEvent，一次 Run 可能产生几千条事件。
数据库能存。
但人没法看。
Run 详情页也会变得很乱。
所以后来我做了事件合并。
不再逐 token 落库，而是按长度或句子边界聚合。
这样前端仍然可以保持流式体验。
数据库里也能保留关键过程。
Run 详情页也重新变得可读。
这一步让我更清楚地区分了两类需求：
实时展示需要细粒度
事后复盘需要可读性
前端看流式输出，可以细一点。
数据库落库和 Run 详情页，则要更适合人阅读。
接入模型不是结束，而是开始
这一阶段做下来，我最大的感受是：
接入大模型不是把 prompt 发过去就结束。
在一个 Agent 项目里，模型只是执行链的一部分。
它要参与决策，要调用工具，要接收工具结果，还要生成最终答案。
真正麻烦的地方不在于“怎么请求模型”，而在于：
模型什么时候该说话
什么时候该调用工具
服务端怎么约束它
工具结果怎么回填
过程怎么展示
失败怎么排查
以前我做前端，更多关注的是接口返回什么，页面怎么展示。
这一次，我开始真正意识到：
AI 应用里的前端和服务端，不只是围着一个接口转。
它们要一起承接一个不断推进的任务过程。
这也是我想复刻公司 Agent 架构的原因。
不是为了做一个聊天框。
而是为了理解：
一个 Agent 是怎么被执行的
模型是怎么参与决策的
Tool 是怎么被安全调用的
Skill 是怎么被编排的
中间过程是怎么被展示的
失败问题是怎么被复盘的
从 Mock Agent 到真实模型接入，这一步让我感觉项目进入了新的阶段。
前面做的是：
流程跑通
现在开始做的是：
让真实模型参与流程
而一旦真实模型进入系统，Agent 就不再只是代码里的状态机。
它开始变成一个需要被约束、被观察、被复盘的执行系统。
这一步做完后，Agent Console 才真正从模拟执行，走向了真实模型调用。
````
