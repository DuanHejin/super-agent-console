# 部署跑通之后，我才真正开始做 Agent

前几天一直在折腾部署。

买服务器、配域名、装 K3s、写 Dockerfile、推 GHCR、接 GitHub Actions、配置 ConfigMap 和 Secret、接 CLS 日志采集。

这些事情做完之后，项目终于有了一条完整的上线链路。

代码合并到 release 分支之后，GitHub Actions 会自动构建 Docker 镜像，推送到 GHCR，再通过 SSH 到香港服务器上更新 K3s 里的 Pod。

访问域名，能看到页面。

线上日志，也能在腾讯云 CLS 里查到。

到这里，严格来说，一个网站的基础运维层已经跑通了。

但我很快意识到，这还不是我真正想做的东西。

因为我想复刻的，并不是一个普通网站，也不是一个简单的 AI 聊天页面。

我真正想做的是一个轻量版的 Agent 项目。

---

前面几天做部署的时候，我其实有一种很明显的感觉：

这些工作很重要，但它们更像是在铺路。

域名、服务器、K3s、CI/CD、日志、环境变量，这些东西决定了项目能不能上线，能不能稳定运行，出了问题能不能排查。

但它们还没有回答一个更核心的问题：

这个项目本身到底要做什么？

如果只是做一个页面，调用一下大模型接口，把返回内容展示出来，那其实很快就能做完。

一个输入框。
一个发送按钮。
一个接口。
一个展示区。

这当然也是 AI 应用。

但它不是我这次真正想练的东西。

我这次想练的是公司项目里那种 Agent 执行链路：

用户输入之后，系统不是简单把问题丢给模型，而是创建一次任务，让 Agent Runtime 去执行。

执行过程中，模型可能会分析问题，可能会调用工具，工具内部还会编排多个 Skill，每一步都会产生过程数据。

前端不只是等最后答案，而是能看到：

```txt
Agent 开始执行
模型正在分析
模型决定调用工具
Tool 开始执行
Skill 1 开始执行
Skill 1 返回结果
Skill 2 开始执行
Skill 2 返回结果
Tool 返回
模型继续生成最终答案
Agent 执行完成
```

这才是我想复刻的重点。

---

所以在运维链路跑通之后，我重新停下来梳理了一遍项目目标。

我把后续能力拆成了三层：

```txt
MVP 必须层
进阶运行层
高阶平台层
```

MVP 层要解决的是核心链路能不能跑通。

也就是：

```txt
创建会话
创建消息
创建 Agent Run
返回 runId
前端用 runId 建立 SSE 订阅
Agent Runtime 执行
模型分析
Tool 调用
Skill 执行
SSE 推送 AgentEvent
前端 Timeline 展示
Run 详情页复盘
数据库落库
```

这一层不追求生产级完整，但必须能把 Agent 执行过程讲清楚、跑起来、看得见。

进阶层再考虑：

```txt
NSQ 异步任务
Redis SSE Replay
上下文管理
多 Tool 动态调用
Retry / Cancel / Timeout
更完整的状态机
```

高阶层再考虑：

```txt
Nacos 配置中心
Coze Loop / Trace
Tool 和 Skill 配置后台
可视化 Workflow
RAG
Memory
多模型 Adapter
```

这样拆完之后，我反而轻松了一些。

因为如果一开始就想把所有东西都做完，会很容易陷入平台化幻想。

什么都想做，最后什么都做不扎实。

所以第一阶段我只给自己定了一个目标：

先把 Agent 的核心执行链路跑通。

---

这里有一个很重要的取舍：

MVP 阶段我没有先接真实大模型。

一开始我也想直接接豆包或者火山方舟。

但后来想了想，如果模型能力、工具调用、SSE 展示、状态机、数据库落库全部一起做，一旦出问题，很难判断问题到底在哪。

是模型没按预期返回？
是 tool call 格式不对？
是服务端状态没流转？
是 SSE 解析有问题？
还是前端展示没处理好？

所以我先做了一个 mock Agent Runtime。

它不依赖真实模型，但会模拟一次完整 Agent 执行：

```txt
生成 runId / traceId
输出 agent_start
输出 model_text_delta
输出 tool_call_start
执行两个 mock Skill
输出 skill_start / skill_result
输出 tool_call_result
输出 final_answer_delta
输出 agent_done
```

这样做的好处是，整个系统的骨架先稳定下来。

前端可以先验证流式展示。
服务端可以先验证事件协议。
数据库可以先验证落库结构。
Run 详情页可以先验证复盘能力。

等这些都通了，再接真实 LLM，问题会少很多。

---

这个阶段，我给项目新增了三份计划文档。

分别是：

```txt
docs/plans/mvp-agent-core-plan.md
docs/plans/advanced-agent-runtime-plan.md
docs/plans/platform-agent-capability-plan.md
```

它们不是为了写文档而写文档，而是为了控制项目边界。

每做完一步，就在对应的 checklist 里打勾。

这样我能清楚知道，现在到底是在做 MVP，还是又忍不住跑去做高阶平台能力了。

对我这种容易越想越大的项目来说，这个约束很有必要。

---

回头看，从 CLS 日志采集结束到开始做 Agent，本质上是一个阶段切换。

前面做的是：

```txt
项目怎么上线
项目怎么发布
项目怎么配置
项目怎么采集日志
```

现在开始做的是：

```txt
Agent 怎么创建任务
Agent 怎么执行
Agent 怎么调用工具
Agent 怎么展示过程
Agent 怎么复盘
Agent 怎么落库
```

前者是项目的运行底座。

后者才是项目的业务内核。

如果没有前者，项目跑不起来。
如果没有后者，项目没有真正的产品价值。

---

这次我最大的感受是：

做一个 AI 项目，不能只盯着模型接口。

模型当然重要，但一个能运行、能调试、能复盘的 Agent 系统，背后还有很多工程结构。

比如：

```txt
一次请求怎么变成一次 Run
Run 的状态怎么流转
每一步过程怎么变成事件
事件怎么推给前端
前端怎么展示过程
执行完之后怎么复盘
日志和数据库怎么串起来
```

这些东西看起来没有模型那么“智能”，但它们决定了这个 Agent 项目是不是真的可控。

如果只是调用一次模型接口，模型回答错了，我只能说它错了。

但如果我能看到完整执行过程，我至少能判断：

```txt
是 prompt 没写清楚
是模型没有正确选择工具
是工具参数错了
是 Skill 输出不合理
是最终回答阶段偏了
还是前端展示解析错了
```

这就是 Agent Console 的意义。

它不是为了炫技，而是为了让 Agent 的执行过程变得可观察。

---

所以接下来，我正式从运维层进入 Agent MVP 实现层。

第一步，就是把原来的“一个聊天接口”，拆成更接近真实项目的双接口结构：

```txt
POST /api/conversations/messages
GET /api/agent/runs/:runId/events
```

一个负责创建消息和 Run。
一个负责订阅 Run 的执行事件。

这会是下一篇要记录的内容。

因为也是从这一步开始，我才真正感觉到：

我做的不是一个聊天页面。

而是一个 Agent 系统。
