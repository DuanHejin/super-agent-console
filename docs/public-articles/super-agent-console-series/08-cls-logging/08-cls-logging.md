# 接入 CLS 日志：线上问题终于不再只靠猜

前面几步做完后，项目已经能自动发布到 K3S。

配置和密钥也通过 ConfigMap、Secret、GitHub Secrets 分开管理。

这时候项目已经不只是“能跑起来”。

它开始有了一点真实线上服务的样子。

但线上服务还有一个绕不开的问题：

出了问题怎么查？

本地开发时，问题通常很好定位。

页面报错了，看浏览器控制台。

接口报错了，看本地终端。

代码改完，重新启动，再试一次。

但部署到服务器上之后，情况就不一样了。

应用跑在容器里，容器跑在 K3S 里，K3S 又跑在云服务器上。

浏览器里看到一个 500，背后可能是：

```txt
应用代码异常
Pod 没启动
Service 转发错误
Ingress 配置问题
静态资源丢失
环境变量没注入
第三方接口失败
```

如果没有日志，就只能猜。

我不想让这个项目停留在“能访问页面”这个阶段。

既然要练完整产品交付，就必须把日志采集也接起来。

所以这一阶段，我开始接入腾讯云 CLS。

## 为什么接 CLS

CLS 是腾讯云的日志服务。

它可以把服务器或容器里的日志采集到云端，然后在控制台里检索、查询和分析。

我一开始的目标很简单：

```txt
把 K3S 里应用的 stdout 日志采集到 CLS
能在腾讯云控制台搜索到
后续线上报错时，可以靠日志定位问题
```

这一步不是为了做很复杂的日志分析。

也不是一上来就做完整可观测性平台。

只是先解决一个最基础的问题：

线上日志要能集中看。

这对个人项目也很重要。

因为一旦服务部署到云服务器上，就不能再只依赖本地终端了。

## 先准备 CLS 资源

我先在腾讯云后台创建了几个基础资源：

```txt
日志集
日志主题
机器组
采集规则
```

日志集可以理解为一个大的日志容器。

日志主题则是具体存放某类日志的地方。

机器组用来标识哪些服务器要被采集。

采集规则决定从这些服务器上采哪些日志、怎么解析。

这些概念一开始看起来有点多。

但真正梳理后，其实就是：

```txt
哪台机器
采哪个路径
送到哪个日志主题
按什么格式解析
```

这几个问题搞清楚，CLS 配置就没那么绕了。

## 在服务器上安装 LogListener

腾讯云要采集服务器日志，需要在服务器上安装 LogListener。

我在 K3S 所在的香港轻量服务器上安装了 LogListener。

安装过程中，需要配置腾讯云子账号的 SecretId 和 SecretKey。

这里也再次遇到了密钥管理问题。

这些凭证不是应用业务密钥，而是日志采集客户端访问腾讯云的权限。

所以它们也不能写进项目代码里。

配置完成后，LogListener 正常启动。

腾讯云后台的机器组也识别到了这台服务器。

到这里，说明：

```txt
服务器
→ LogListener
→ 腾讯云 CLS
```

这条采集通道已经建立起来了。

## 标准输出到底在哪里

接下来配置采集规则时，我卡了一下。

我想采集的是容器标准输出。

但腾讯云后台要求填写采集路径。

一开始我有点疑惑：

我不是采集 stdout 吗？

为什么还要填文件路径？

后来理清楚后才明白：

在 K3S 里，容器标准输出最终也会落到节点上的日志文件。

`kubectl logs` 能看到的内容，本质上也是从这些容器日志文件读出来的。

常见路径会在：

```txt
/var/log/pods/...
/var/log/containers/...
```

而我当前用的是服务器 LogListener + 机器组的方式。

它不是直接接入 Kubernetes 采集组件，而是从宿主机文件系统里采日志。

所以采集规则里要求填写路径是正常的。

也就是说，我当前这条链路其实是：

```txt
应用 stdout
→ K3S/containerd 写入节点日志文件
→ LogListener 采集文件
→ CLS
```

这件事对我挺有帮助。

因为它让我意识到，“标准输出”并不等于没有文件。

在容器环境里，stdout 也会被 runtime 接管和落盘。

日志系统最终还是要从某个地方把它读出来。

## 提取模式先选单行全文

配置采集规则时，还有一个问题是提取模式。

项目里现在的日志来源不完全一致：

```txt
服务端 pino JSON 日志
Nuxt/Nitro 自己输出的错误日志
一些普通字符串日志
```

这些日志不一定每一行都是完整 JSON。

而且 K3S/containerd 的日志文件里，前面可能还会带一层时间戳、stdout/stderr 标记。

如果直接选择 JSON 提取模式，很容易解析失败。

所以我最后先选择了：

```txt
单行全文
```

这不是最结构化的方式。

但它最稳。

当前阶段，我的目标不是马上做复杂字段分析，而是先确保日志能进 CLS、能搜索、能看到完整报错。

先把采集链路跑通，再慢慢优化日志格式。

这也是我这几天反复感受到的一点：

先通，再优雅。

## 日志采集成功

采集规则配置完成后，我在 CLS 控制台里看到了日志。

这一步对我来说挺关键。

因为它说明从应用容器到腾讯云日志服务的链路已经打通：

```txt
Nuxt 应用 stdout
→ K3S 节点日志文件
→ LogListener
→ CLS 日志主题
→ 控制台检索
```

到这里，线上问题就不再只能靠猜了。

至少我可以在 CLS 里看到服务端到底输出了什么。

以前做前端时，线上报错很多时候是看浏览器控制台，或者等后端、运维同事查日志。

现在我自己把这条链路接起来之后，对线上服务的“可观察性”有了更具体的感觉。

日志不是锦上添花。

它是线上系统的眼睛。

没有日志，很多问题只能靠猜。

## 一次真实排障

日志刚接上没多久，就真的派上用场了。

我在 CLS 里看到一批报错：

```txt
/_nuxt/*.js 500
ENOENT: no such file or directory
```

大概意思是，浏览器请求 Nuxt 的静态资源时，服务端找不到对应文件。

页面 HTML 能返回，但 JS/CSS 加载失败。

如果没有 CLS，我可能只会在浏览器里看到页面异常，然后继续猜：

```txt
是不是 Ingress 配错了
是不是 Service 端口不对
是不是镜像没更新
是不是缓存问题
```

但日志里已经给出了非常明确的路径：

```txt
/app/.output/server/chunks/public/_nuxt/xxx.js
```

它说明 Nitro 在这个路径下找静态资源。

而我检查构建产物后发现，Nuxt 实际生成的静态资源在：

```txt
/app/.output/public/_nuxt
```

也就是说，Docker 镜像里的静态资源布局和 Nitro 运行时查找路径不一致。

最后我修改 Dockerfile，把 `.output/public` 同步到 Nitro 查找的路径：

```txt
.output/server/chunks/public
```

重新构建、发布之后，`/_nuxt/*.js` 的 500 消失了。

这次排障让我非常明显地感受到：

日志采集不是“以后再做”的东西。

它会直接影响你解决线上问题的速度。

## 服务端日志要继续规范

虽然 CLS 已经能采集日志，但我也看到当前日志还不够理想。

现在有几类日志混在一起：

```txt
应用自己的 pino JSON 日志
Nuxt/Nitro 框架日志
普通字符串日志
错误堆栈
```

所以当前先用单行全文是合理的。

但后面如果项目继续发展，还是需要进一步规范服务端日志。

比如每条关键日志都应该带上：

```txt
eventType
traceId
runId
service
env
message
```

尤其这个项目后面要做 Agent 执行链路。

一次 Agent Run 可能会经过：

```txt
前端提交
服务端创建 run
调用模型
工具执行
SSE 推送
数据库落库
前端 Timeline 展示
```

如果没有 `runId` 和 `traceId`，后面排查一次执行问题会很痛苦。

所以 CLS 只是第一步。

它先把日志采上来了。

后面还要让日志本身更有结构。

## 当前结果

这一阶段完成后，我得到的结果是：

```txt
腾讯云 CLS 日志集创建完成
日志主题创建完成
机器组创建完成
K3S 云服务器安装 LogListener
LogListener 使用子账号 SecretId / SecretKey 初始化成功
腾讯云后台识别到服务器
采集规则配置完成
K3S 容器 stdout 日志成功进入 CLS
提取模式先使用单行全文
可以在 CLS 控制台检索线上日志
通过 CLS 发现并修复 /_nuxt/*.js 500 问题
```

也就是说，项目终于有了一个基本的线上排障闭环。

以前是：

```txt
线上出问题
→ 打开浏览器看现象
→ 猜问题在哪
```

现在至少变成了：

```txt
线上出问题
→ 查 CLS 日志
→ 定位错误路径和堆栈
→ 修复代码或镜像
→ 通过 CI/CD 发布
→ 再观察日志
```

这已经很不一样了。

## 给前端和初学者的一点建议

如果你也在做类似练习，我现在会建议：

第一，日志采集不要等到项目很复杂才做。

哪怕只是个人项目，能集中查看线上日志，也会极大降低排障成本。

第二，先用单行全文跑通链路。

不要一开始就纠结复杂 JSON 提取。先确保日志能采集、能搜索、能看到完整错误。

第三，理解容器 stdout 的落地路径。

在 K3S/containerd 里，stdout 最终会变成节点上的日志文件。用 LogListener 采集时，采的其实是这些文件。

第四，日志里不要输出敏感信息。

API Key、数据库密码、token，都不应该出现在 stdout 里。

第五，后续要补结构化字段。

等主链路稳定后，再逐步统一 JSON 日志，补 `traceId`、`runId`、`eventType`。

这就是我重新部署自己的第七步。

我把线上日志接入了腾讯云 CLS。

它不是让项目看起来更高级。

而是让我第一次有了一个真实的线上排障闭环。

从域名、服务器、K3S、Docker、GHCR、GitHub Actions，到 ConfigMap、Secret、CLS 日志。

这一轮下来，我不敢说自己已经掌握了云原生。

但至少我不再只站在浏览器这一侧看问题。

我开始能从代码一路看到容器、部署、配置和日志。

这也是我这次“重新部署自己”真正想补齐的东西。
