# 把 Nuxt3 项目装进 Docker：前端项目开始变成一个可部署单元

前两篇做完后，我已经有了一台可以访问的服务器，也在上面跑通了 K3S。

这时候，公网流量已经可以走到 K3S 里面：

```txt
浏览器
→ 域名
→ K3S Ingress
→ Service
→ Pod
```

不过当时 K3S 里跑的还只是一个 Nginx Demo。

它的作用是验证链路。

真正要部署的，还是我自己的项目。

所以接下来要做的事情，就是创建一个 Nuxt3 项目，并把它做成 Docker 镜像。

这一步对我来说很关键。

以前做前端项目，更多是：

```txt
npm install
npm run dev
npm run build
```

build 出来的产物，可能交给运维，也可能上传到某个静态资源服务器，或者接入公司的发布平台。

但这次不一样。

我需要自己把前端项目变成一个可以被服务器拉取、启动、运行的部署单元。

也就是 Docker 镜像。

## 为什么选 Nuxt3

这个项目的目标，是做一个 AI Agent 控制台。

它不会只是一个纯静态页面。

后面会有：

```txt
服务端接口
模型调用
SSE 流式响应
Agent 执行过程展示
数据库落库
日志输出
部署和排障
```

所以我没有选择普通 Vue 单页应用，而是选了 Nuxt3。

Nuxt3 对我来说有几个好处：

```txt
前端还是 Vue 体系，比较熟悉
内置 Nitro server，可以写 server/api
既可以做页面，也可以做服务端接口
后面适合承接 AI 调用和中转逻辑
```

这也符合我这次转型练习的目标。

不是只做一个页面，而是从前端出发，逐渐补齐服务端和部署能力。

## 创建项目骨架

一开始我先创建了一个最小 Nuxt3 项目。

项目里先不急着做复杂业务，只保留几个基础部分：

```txt
首页
部署说明页
基础组件目录
composables 目录
server/api 目录
server/services 目录
Prisma 目录
docs 目录
```

我还补了几个基础接口：

```txt
/api/health
/api/ready
/api/db-check
```

这些接口一开始看起来很简单，但它们很重要。

因为项目后面部署到 K3S 之后，需要有办法判断服务是否活着、是否准备好、数据库是否能连上。

以前做前端时，我对这些接口的感知不强。

但当你开始自己部署服务时，就会发现：

“页面能打开”并不等于“服务状态健康”。

服务需要有健康检查。

部署平台也需要知道什么时候可以把流量打进来。

所以即使现在只是 Demo，我也先把这些入口留出来。

## Node 版本踩坑

项目初始化后，我遇到的第一个问题不是业务代码，而是 Node 版本。

执行安装和构建时，报了一个错误：

```txt
The requested module 'node:util' does not provide an export named 'styleText'
```

这个错误的原因是当前 Node 版本太低。

Nuxt 相关依赖里用到了比较新的 Node API，而我当时本机默认还是 Node 18。

最后我把项目统一到了 Node 22，并且在项目里补了 `.nvmrc` 和 `package.json engines`。

这样后面每次进入项目，都能明确知道应该使用哪个 Node 版本。

这件事对我也算一个提醒：

前端项目的“能跑”，并不只取决于代码。

Node 版本、包管理器、依赖版本，这些都是工程环境的一部分。

如果这些不固定，今天我能 build，明天换台机器可能就不行。

## 写 Dockerfile

Nuxt3 本地能 build 之后，下一步就是写 Dockerfile。

这一步我一开始的目标很简单：

```txt
在 Docker 里安装依赖
构建 Nuxt 项目
拷贝生产产物
启动 Nitro server
```

最终思路是多阶段构建：

```txt
builder 阶段：
  npm ci
  npm run build

runner 阶段：
  只保留 .output
  node .output/server/index.mjs
```

这样做的好处是，最终运行镜像里不需要保留完整源码和开发依赖。

Dockerfile 写完后，项目就不再只是一个“本地可以运行的前端项目”。

它开始变成一个可以被容器环境运行的应用。

这一步对我来说挺有感觉。

以前写前端，很多时候 build 结束就算交付了一半。

但现在我开始意识到：

真正的交付物不是源码，也不是本地 build 产物，而是一个能在目标环境启动的镜像。

## 本地安装 Docker 环境

要构建镜像，本地还需要 Docker 环境。

我本机是 Mac，所以装了 Docker CLI 和 Colima。

这里也踩过一个小坑。

我之前机器上曾经有过 x86 版本的 Homebrew，导致安装路径和架构有些混乱。

后来重新整理成了原生 arm64 的 Homebrew，再安装 Docker 和 Colima。

这类问题以前我可能会觉得有点烦。

但这次我反而觉得，它也是工程实践的一部分。

因为真实开发环境从来不是完全干净的。

机器上装过什么、终端是不是 Rosetta、Homebrew 是 arm64 还是 x86，这些都会影响后面的工具链。

最后 Docker 环境跑起来后，我就可以在本地执行镜像构建。

## 构建 linux/amd64 镜像

我的服务器是 Linux 环境，所以构建镜像时，需要关注平台。

本地 Mac 是 arm64，但服务器通常是 amd64。

所以构建时我指定了：

```txt
linux/amd64
```

这一步是为了避免本地构建出 arm64 镜像，结果服务器拉下来跑不起来。

这个问题很容易被忽略。

尤其现在很多人用的是 Apple Silicon Mac，本地架构和服务器架构不一样。

如果不注意镜像平台，部署时可能会遇到很奇怪的问题。

最终，我本地成功构建出了项目镜像。

这时候项目已经完成了一个关键变化：

```txt
Nuxt3 源码
→ Docker 镜像
```

## 第一次部署自己的镜像

镜像构建完成后，我最初的方式还比较手动。

先把镜像推送到 GitHub Container Registry。

然后到 K3S 上，把原来 Nginx Demo 的 Deployment image 改成我自己的镜像。

也就是把：

```txt
nginx
```

替换成：

```txt
ghcr.io/xxx/super-agent-console:某个版本
```

同时还要注意端口。

Nginx 默认监听 80。

但 Nuxt Nitro server 监听的是 3000。

所以 Deployment 的 containerPort 和 Service 的 targetPort，都需要改成 3000。

这也是一个很典型的部署细节。

页面本地能跑，不代表放进 Kubernetes 就能访问。

容器监听端口、Service 转发端口、Ingress 指向的 Service，都要对齐。

这些以前对我来说更多是“运维配置”。

但这次自己改了一遍之后，才真正把它们串起来。

## 访问正式项目首页

更新镜像后，我重新访问自己的域名。

这一次，页面不再是 Nginx 默认页。

而是我自己的 Nuxt 项目首页。

这个结果对我来说很重要。

因为它说明完整链路已经从 Demo 走到了真实项目：

```txt
浏览器
→ 域名
→ K3S Ingress
→ Service
→ Nuxt Pod
→ Nuxt 首页
```

虽然这个首页还很简单，业务功能也还没开始做。

但它已经不是本地项目了。

它已经作为一个 Docker 镜像，被部署到了我自己的 K3S 里，并且可以通过公网域名访问。

这一步让我对“部署”这件事的感受更具体了。

以前我可能会说：

前端项目上线了。

现在我更愿意把它拆开看：

```txt
代码能构建
镜像能生成
镜像能推送
服务器能拉取
Pod 能启动
Service 能转发
Ingress 能接流量
浏览器能访问
```

每一步都可能出问题。

每一步也都值得被理解。

这一步对我来说，是前端能力边界的一次外扩。

以前我更关心页面怎么写。

现在我开始关心：

```txt
这个页面如何作为服务运行起来
```

这些不是为了让我变成专职运维。

## 给前端和初学者的一点建议

如果你也在做类似练习，我现在会建议：

第一，先让项目本地 build 成功，再写 Dockerfile。

不要一边解决业务问题，一边解决 Docker 问题。先把变量减少。

第二，固定 Node 版本。

`.nvmrc` 和 `engines` 很有用。很多奇怪错误，其实只是运行环境不一致。

第三，Apple Silicon Mac 要注意镜像平台。

如果服务器是 amd64，构建时要显式指定 `linux/amd64`。

第四，Deployment 和 Service 的端口要对齐。

Nginx 默认 80，Nuxt 默认 3000。换镜像时不要只改 image，端口也要一起检查。

第五，第一次可以手动部署。

不要一开始就追求 CI/CD 自动化。先手动跑通，理解每一步发生了什么，再把它自动化。

这就是我重新部署自己的第三步。

我把一个 Nuxt3 前端项目，装进了 Docker 镜像里。

从这一刻开始，它不再只是本地代码。

它变成了一个可以被服务器拉取、启动和替换的部署单元。
