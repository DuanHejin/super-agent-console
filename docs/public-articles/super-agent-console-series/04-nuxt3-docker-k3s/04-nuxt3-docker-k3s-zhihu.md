标题
前端项目不只是 npm run build：我把 Nuxt3 项目第一次装进了 Docker
正文
前两篇里，我已经完成了两件事。
第一步，买域名和香港轻量应用服务器，让一个域名真正访问到自己的服务器。
第二步，在服务器上安装 K3S，把公网入口从宿主机 Nginx 切换到了 K3S Ingress。
到这里，公网流量已经可以进入 K3S：

浏览器
→ 域名
→ K3S Ingress
→ Service
→ Pod

不过当时 K3S 里跑的还只是一个 Nginx Demo。
它的作用是验证链路，不是最终项目。
真正要部署的，是我自己的 Nuxt3 项目。
所以接下来要做的事情，就是：
创建 Nuxt3 项目，并把它打成 Docker 镜像，部署到自己的 K3S 里。
这一步对我来说很关键。
因为以前做前端项目，我更熟悉的是：

npm install
npm run dev
npm run build

build 出来的产物，可能交给运维，也可能上传到静态资源服务器，或者进入公司的发布平台。
但这次不一样。
我需要自己把前端项目变成一个可以被服务器拉取、启动、运行和替换的部署单元。
也就是 Docker 镜像。

为什么这个项目选 Nuxt3？
这个项目的目标，是做一个 AI Agent 控制台。
它不会只是一个纯静态页面。
后面会有：

服务端接口
模型调用
SSE 流式响应
Agent 执行过程展示
数据库落库
日志输出
部署和排障

所以我没有选择普通 Vue 单页应用，而是选择了 Nuxt3。
原因很简单。
Nuxt3 仍然是 Vue 体系，我比较熟悉；同时它内置 Nitro server，可以在 server/api 里写接口。
这意味着它既能做页面，也能承接一些服务端逻辑。
对我这个阶段来说，这很合适。
我不是想只做一个页面，而是想从前端出发，逐渐补齐服务端、AI 调用、部署和日志排障能力。
所以这次的技术路线大概是：

Nuxt3 页面

- Nitro Server API
- Docker 镜像
- K3S 部署
- 后续 Agent / SSE / 日志链路

先创建一个最小项目骨架
一开始我没有急着做复杂业务。
先创建了一个最小 Nuxt3 项目，并预留了一些基础目录：

首页
部署说明页
基础组件目录
composables 目录
server/api 目录
server/services 目录
Prisma 目录
docs 目录

同时补了几个基础接口：

/api/health
/api/ready
/api/db-check

这些接口看起来很简单，但它们很重要。
以前做前端时，我对这类接口感知不强。
但当自己开始部署服务之后，就会发现：
页面能打开，不代表服务一定是健康的。
服务需要有健康检查。
部署平台也需要知道什么时候可以把流量打进来。
比如：

/api/health：服务是否活着
/api/ready：服务是否准备好接流量
/api/db-check：数据库是否连接正常

即使现在只是一个 Demo，我也先把这些入口留出来。
因为后面进入 K3S 部署、日志采集、Agent 执行链路之后，这些能力都会用得上。

第一个坑：Node 版本
项目初始化之后，我遇到的第一个问题不是业务代码，而是 Node 版本。
安装和构建时，报了一个错误：

The requested module 'node:util' does not provide an export named 'styleText'

最后发现，是当前 Node 版本太低。
Nuxt 相关依赖里用到了比较新的 Node API，而我本机默认还是 Node 18。
于是我把项目统一到了 Node 22，并且补了两个配置：

.nvmrc
package.json engines

这样后面每次进入项目，都能明确知道应该使用哪个 Node 版本。
这件事给我的提醒是：
前端项目的“能跑”，不只取决于代码。
Node 版本、包管理器、依赖版本，这些都是工程环境的一部分。
如果这些不固定，今天我能 build，明天换台机器可能就不行。

开始写 Dockerfile
Nuxt3 本地能 build 之后，下一步就是写 Dockerfile。
一开始目标很简单：

在 Docker 里安装依赖
构建 Nuxt 项目
拷贝生产产物
启动 Nitro server

整体采用多阶段构建：

builder 阶段：
npm ci
npm run build

runner 阶段：
只保留 .output
node .output/server/index.mjs

这样做的好处是，最终运行镜像里不需要保留完整源码和开发依赖。
Dockerfile 写完后，项目就不再只是一个“本地可以运行的前端项目”。
它开始变成一个可以被容器环境运行的应用。
这一步对我很有感觉。
以前做前端，很多时候 npm run build 成功，就觉得已经完成了很大一部分。
但现在我开始意识到：
真正的交付物不是源码，也不只是 build 产物，而是一个能在目标环境启动的镜像。

本地 Docker 环境也会踩坑
要构建镜像，本地还需要 Docker 环境。
我本机是 Mac，所以装了 Docker CLI 和 Colima。
这里也遇到过一个小问题。
我之前机器上曾经有过 x86 版本的 Homebrew，导致安装路径和架构有些混乱。
后来重新整理成原生 arm64 Homebrew，再安装 Docker 和 Colima。
这类问题以前我可能会觉得有点烦。
但这次反而觉得，它也是工程实践的一部分。
真实开发环境从来不是完全干净的。
机器上装过什么、终端是不是 Rosetta、Homebrew 是 arm64 还是 x86，这些都会影响工具链。
最后 Docker 环境跑起来后，我就可以在本地执行镜像构建了。

Apple Silicon Mac 要注意镜像平台
我的本地电脑是 Apple Silicon Mac，是 arm64 架构。
但服务器是 Linux amd64。
所以构建镜像时，我需要显式指定：

linux/amd64

这一步是为了避免本地构建出 arm64 镜像，结果服务器拉下来跑不起来。
这个问题很容易被忽略。
尤其现在很多前端开发都用 Mac，本地架构和服务器架构不一致。
如果不注意镜像平台，部署时可能会遇到一些看起来很奇怪的问题。
最终，我本地成功构建出了项目镜像。
这时候项目完成了一个关键变化：

Nuxt3 源码
→ Docker 镜像

第一次部署自己的镜像
镜像构建完成后，我最开始采用的是比较手动的方式。
先把镜像推送到 GitHub Container Registry。
然后到 K3S 上，把原来 Nginx Demo 的 Deployment image 改成我自己的镜像。
也就是把：

nginx

替换成：

ghcr.io/xxx/super-agent-console:某个版本

同时还要注意端口。
Nginx 默认监听 80。
但 Nuxt Nitro server 监听的是 3000。
所以 Deployment 的 containerPort 和 Service 的 targetPort，都需要改成 3000。
这是一个很典型的部署细节。
页面本地能跑，不代表放进 Kubernetes 就能访问。
容器监听端口、Service 转发端口、Ingress 指向的 Service，都要对齐。
以前这些对我来说更多是“运维配置”。
这次自己改了一遍后，才真正把它们串起来。

看到自己的 Nuxt 首页在线上打开
更新镜像后，我重新访问自己的域名。
这一次，页面不再是 Nginx 默认页。
而是我自己的 Nuxt 项目首页。
这个结果对我来说很重要。
因为它说明完整链路已经从 Demo 走到了真实项目：

浏览器
→ 域名
→ K3S Ingress
→ Service
→ Nuxt Pod
→ Nuxt 首页

虽然首页还很简单，业务功能也还没开始做。
但它已经不是本地项目了。
它已经作为一个 Docker 镜像，被部署到了自己的 K3S 里，并且可以通过公网域名访问。
这一步让我对“部署”有了更具体的理解。
以前我可能会简单地说：
前端项目上线了。
现在我更愿意拆开看：

代码能构建
镜像能生成
镜像能推送
服务器能拉取
Pod 能启动
Service 能转发
Ingress 能接流量
浏览器能访问

每一步都可能出问题。
每一步也都值得理解。

当前阶段完成了什么？
这一阶段完成后，我得到的结果是：

创建了 Nuxt3 项目
确认本地 dev 和 build 可用
解决了 Node 版本问题
编写了 Dockerfile
安装并跑通本地 Docker 构建环境
构建了 linux/amd64 镜像
手动推送镜像到 GHCR
在 K3S 中把 Nginx Demo 镜像替换为 Nuxt 项目镜像
通过域名访问到了正式项目首页

也就是说，项目完成了从：

本地前端代码

到：

线上容器应用

的第一次转换。
这一步对我来说，是前端能力边界的一次外扩。
以前我更关心页面怎么写。
现在我开始关心：

这个页面如何作为服务运行起来
如何被容器承载
如何被 Kubernetes 调度
如何被公网流量访问到

这些不是为了让我变成专职运维。
而是为了以后做 AI / Agent 产品时，能更完整地理解一个产品从代码到上线的过程。

给前端和初学者的一点建议
如果你也在做类似练习，我现在会建议：
第一，先让项目本地 build 成功，再写 Dockerfile。
不要一边解决业务问题，一边解决 Docker 问题。先减少变量。
第二，固定 Node 版本。
.nvmrc 和 engines 很有用。很多奇怪错误，其实只是运行环境不一致。
第三，Apple Silicon Mac 要注意镜像平台。
如果服务器是 amd64，构建时要显式指定 linux/amd64。
第四，Deployment 和 Service 的端口要对齐。
Nginx 默认 80，Nuxt 默认 3000。换镜像时不要只改 image，端口也要一起检查。
第五，第一次可以手动部署。
不要一开始就追求 CI/CD 自动化。先手动跑通，理解每一步发生了什么，再把它自动化。

最后
这是我重新部署自己的第三步。
我把一个 Nuxt3 前端项目，装进了 Docker 镜像里。
从这一刻开始，它不再只是本地代码。
它变成了一个可以被服务器拉取、启动和替换的部署单元。

如何把手动构建和手动部署，改造成 GitHub Actions 自动发布流程。
