# 重新部署自己的第一步：先买一套能低成本试错的云环境

决定做这个 AI Agent 控制台之后，我没有第一时间写代码。

以前做前端项目，我的习惯是先建项目、拆页面、写组件、调接口。页面能跑起来，好像项目就开始了。

但这次我想换一个顺序。

我想先从最外层开始：买一个域名，买一台服务器，让浏览器能通过域名访问到它。

这件事不复杂，但对我来说有意义。

因为过去很多时候，服务器、域名、网关、部署环境都是别人准备好的。前端工程师更多是在一个已经搭好的系统里工作。

而我现在想练的是从一个想法到一个可运行产品的完整链路。

所以第一步，不是写页面，而是先搭一个低成本、能试错、后面还能继续扩展的基础环境。

## 我的选型原则

这次选型没有追求完美。

核心就是几个很实际的要求：

```txt
不用备案
成本低一点
后面能练 Docker / K3S / Ingress
如果项目继续做，也不用推倒重来
```

因为这个项目一开始就是学习和测试为主，我不想把太多时间花在备案流程上，也不想一开始就买很贵的资源。

最后确定的组合是：

```txt
腾讯云香港轻量应用服务器
一个低成本 .top 域名
```

不算高配，但足够我先跑起来。

## 服务器选择

服务器我最后选的是腾讯云香港轻量应用服务器。

配置大概是：

```txt
2 核 CPU
2G 内存
40GB SSD
每月 0.5TB 流量
Ubuntu 22.04
中国香港
38 元 / 月
```

选香港主要是为了免备案。

域名解析过去之后，可以直接访问，不用等十几天备案流程。对一个练习项目来说，这点很重要。

配置上，我没有选 1 核 2G，而是选了 2 核 2G。

原因也很简单：后面要跑 K3S。

1 核也许不是不能跑，但余量会比较紧。我不想刚开始练习云原生，就把时间耗在资源不够导致的各种小问题上。

这个配置跑 Nginx、K3S、Ingress、Nuxt 应用和一些测试服务，前期基本够用。

不过这里也有一个后来才意识到的点：

轻量应用服务器虽然便宜好用，但它不是 CVM。

这会影响后面使用腾讯云 TKE 的方案。也正是因为这个选择，我后面才从原计划的 TKE 改成了自己安装单节点 K3S。

这算是这次选型里的一个小缺点。

当时只想着轻量、便宜、免备案，后来才发现云产品之间并不是完全打通的。

## 域名选择

域名这块，我一开始也纠结过要不要买 `.com`。

`.com` 确实更正式，但对一个练习项目来说，有点没必要。

我最后选了 `.top`。

一年十几块钱，成本很低，解析到香港服务器也不用备案。前期用来做项目练习、环境测试、功能调试，完全够用。

如果项目后面真的做起来，再买一个正式一点的 `.com` 域名也来得及。

到时候只要把新域名解析到现有服务器，整体架构不用重来。

这里顺便记录一个小知识点：

买了一个顶级域名之后，二级域名是不需要额外付费的。

比如：

```txt
api.xxx.top
admin.xxx.top
demo.xxx.top
```

这些都可以在 DNS 控制台里免费添加解析记录。

这个对后面做项目会很方便，因为不同服务可以挂在不同二级域名下面。

## 第一条链路：域名访问服务器

服务器和域名买好之后，我做的第一件事就是配置 DNS。

核心就是添加一条 A 记录：

```txt
域名 -> 服务器公网 IP
```

解析生效后，就可以通过域名找到这台服务器。

但这还不够。

服务器上还得有服务接住这个请求。

所以我先安装了 Nginx。

这一步没有什么复杂配置，就是为了验证：

```txt
浏览器
→ 域名 DNS
→ 服务器公网 IP
→ 服务器 80 端口
→ Nginx
```

这条链路是不是通的。

安装并启动 Nginx 后，我在浏览器里访问自己的域名，看到了 Nginx 默认欢迎页。

这页本身没什么价值。

但对我来说，它代表第一条公网访问链路打通了。

以前说“页面上线”，我更多关注的是前端代码有没有构建成功。

这次自己从域名和服务器开始搭，感受会更具体一点：

一个页面能被访问到，背后不是只有代码。

DNS、服务器、公网 IP、端口、Web 服务，这些都得一起正常工作。

任何一个环节不通，浏览器里看到的就不是页面，而是超时、拒绝连接、解析失败，或者 502。

## 一些不需要急着买的东西

这次下单和配置过程中，我也顺手确认了几个容易让人多花钱的点。

付费 DNS 暂时没必要。

域名注册后，一般会提供基础 DNS 解析服务。个人练习项目够用了。

付费 SSL 证书也没必要一开始买。

后面需要 HTTPS 时，可以用 Let’s Encrypt 免费证书。个人项目完全够用。

服务器下单时附带的一些备份、安全加固、付费监控，也可以先不选。

这些东西对生产环境当然有价值，但我当前只是学习和验证，先控制成本更重要。

这也是这次项目的一个原则：

先跑通，再补强。

不要一开始就把自己困在一堆“看起来很专业”的选项里。

## 这一阶段的结果

第一阶段完成后，我得到的结果很简单：

```txt
买好了域名
买好了香港轻量应用服务器
DNS 解析到了服务器公网 IP
服务器上安装了 Nginx
浏览器访问域名，可以看到 Nginx 默认页面
```

没有写业务代码，也没有接 AI。

但我确认了一个基础事实：

我已经有了一个自己的公网入口。

后面不管是部署 Nuxt 应用，还是接入 K3S、Ingress、CI/CD，本质上都要从这个入口继续往里搭。

对一个前端来说，这一步可能不是最难的技术点。

但它是一个边界。

过去我更习惯站在浏览器这一侧，关心页面如何呈现。

这次我开始往浏览器后面多走几步，去看请求到底经过了什么，服务器怎么接住它，应用又应该怎么部署上去。

## 给前端和初学者的一点建议

如果你也想搭一个自己的练习环境，我现在会建议：

第一，不要一开始就追求完美方案。

练习项目最重要的是先跑起来，而不是第一天就设计出生产级架构。

第二，香港轻量服务器适合低成本试错。

免备案、成本低、上手快，对个人练习很友好。但也要知道，它和 CVM 不完全一样，后面接云厂商托管 K8S 时可能会有限制。

第三，域名不用一开始就买贵的。

`.top`、`.online` 这类低成本域名，完全可以支撑前期学习和测试。

第四，少买增值服务。

备份、安全加固、付费监控、付费证书这些，不是不重要，而是可以后置。

第五，先落地，再优化。

先让域名能访问服务器，再让服务器跑应用，再让应用进入容器，最后再考虑 K3S、CI/CD 和日志采集。

这就是我重新部署自己的第一步。

不是先写一个炫酷页面。

而是先用低成本搭出一个真实入口，让一个域名，真正访问到我自己的服务器。

下一篇，我会记录这次计划里的第一次变化：

我原本以为可以直接使用腾讯云 TKE，但后来发现轻量应用服务器并不支持这条路。

于是，部署方案从托管 Kubernetes，改成了自己在服务器上安装单节点 K3S。
