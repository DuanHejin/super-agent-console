标题
Nuxt3 AI Agent 控制台实战 01：低成本云环境准备，先让域名访问到自己的服务器
正文
这是我做 Super Agent Console 的第一篇记录。
这个项目的目标是做一个基于 Nuxt3 的 AI Agent 工程化 Demo，后续会逐步覆盖：
Nuxt3 全栈开发
Docker 镜像构建
K3S 单节点部署
Ingress 域名访问
GitHub Actions 自动部署
ConfigMap / Secret 环境配置
CLS 日志采集
SSE 流式输出
Agent Timeline
工具调用展示
但项目开始的第一步，我没有直接写代码。
而是先买域名、买服务器，验证最基础的公网访问链路。
也就是：
Browser
→ Domain
→ DNS
→ Server IP
→ Nginx
这个动作看起来很简单，但对一个想补齐产品交付链路的前端来说，是一个很好的起点。

1. 为什么不先写代码？
   以前做前端项目，我一般会先做这些事情：
   初始化项目
   拆页面
   写组件
   接接口
   跑本地开发环境
   页面在本地能跑起来，就感觉项目已经开始了。
   但这次我想换一个顺序。
   因为这个项目的目标不是单纯做一个页面，而是想完整练一遍：
   代码
   → 镜像
   → 容器
   → Kubernetes
   → 域名访问
   → 日志采集
   → 线上排障
   所以第一步我选择从最外层开始：
   先准备一个低成本、可访问、可继续扩展的云环境。

2. 云环境选型原则
   这次选型没有追求高配置，也没有追求企业级标准。
   核心原则只有几个：
   不用备案
   成本低
   能跑基础服务
   后续能练 Docker / K3S / Ingress
   前期不需要推倒重来
   最终选择：
   腾讯云香港轻量应用服务器
   低成本 .top 域名
   这套组合的特点是：
   便宜
   上手快
   免备案
   适合个人练习项目

3. 服务器配置
   服务器选择的是腾讯云香港轻量应用服务器。
   配置大概如下：
   CPU：2 核
   内存：2G
   硬盘：40GB SSD
   流量：0.5TB / 月
   系统：Ubuntu 22.04
   地域：中国香港
   价格：38 元 / 月
   这里没有选中国内地服务器，主要是为了免备案。
   对于练习项目来说，如果第一步就进入备案流程，会拉长启动周期。
   我希望这个项目可以快速验证：
   域名是否能访问服务器
   服务器是否能跑服务
   后续是否能承载 K3S 和 Nuxt 应用
   所以香港轻量服务器更适合当前阶段。

4. 为什么选 2 核 2G？
   一开始我也考虑过 1 核 2G。
   但后面计划在服务器上安装 K3S，所以最终选了 2 核 2G。
   原因很简单：
   1 核可以省钱，但余量太小
   K3S + Ingress + Nuxt 应用 + 日志采集会占用一定资源
   不想把时间浪费在资源过低导致的排障上
   这个配置不是生产级别，但用于个人 Demo 前期验证基本够用。

5. 轻量应用服务器的一个限制
   这里有一个后面才意识到的问题。
   轻量应用服务器不是 CVM。
   我原本的计划是：
   Docker 镜像
   → 腾讯云 TCR
   → 腾讯云 TKE
   → Pod 部署
   但后来发现，腾讯云 TKE 不能直接使用我购买的轻量应用服务器作为节点。
   所以后续方案调整成了：
   Docker 镜像
   → GitHub Container Registry
   → 自建 K3S 单节点集群
   → Deployment / Ingress 部署应用
   这算是这次选型里的一个限制。
   但从学习角度看，反而也有价值。
   因为我可以自己完整走一遍：
   安装 K3S
   创建 Deployment
   创建 Service
   配置 Ingress
   通过域名访问 Pod

6. 域名选择
   域名方面，我没有买 .com，而是选了一个低成本 .top 域名。
   原因很简单：
   价格低
   不需要一开始追求正式品牌感
   适合练习和测试
   后续可以替换为更正式域名
   前期这个域名主要用于：
   访问测试
   Ingress host 配置
   Nuxt 应用部署
   API 子域名预留
   日志和回调地址测试
   另外，一个顶级域名下面可以免费添加多个二级域名解析。
   例如：
   api.xxx.top
   admin.xxx.top
   demo.xxx.top
   agent.xxx.top
   后面如果服务拆分，可以直接通过不同二级域名区分。

7. 配置 DNS 解析
   域名和服务器准备好后，第一步就是配置 DNS。
   添加一条 A 记录：
   主机记录：@
   记录类型：A
   记录值：服务器公网 IP
   如果需要二级域名，也可以添加：
   主机记录：demo
   记录类型：A
   记录值：服务器公网 IP
   配置完成后，等待解析生效。
   这一步完成后，域名已经能解析到服务器公网 IP。
   但浏览器访问域名时，还需要服务器上有进程监听对应端口。
   所以我先用 Nginx 做验证。

8. 安装 Nginx 验证公网访问链路
   在服务器上安装 Nginx 后，启动服务。
   目的不是正式部署 Nginx，而是验证链路：
   浏览器
   → 域名
   → DNS 解析
   → 服务器公网 IP
   → 80 端口
   → Nginx 默认页
   当浏览器访问域名，看到 Nginx 默认欢迎页时，就说明第一条链路打通了。
   这一步虽然简单，但很关键。
   因为后面无论是 K3S Ingress，还是 Nuxt 应用，最终都要走这个入口。

9. 这一步验证了什么？
   看到 Nginx 默认页后，至少说明以下几件事是正常的：
   域名购买成功
   DNS 解析生效
   服务器公网 IP 可访问
   服务器安全组 / 防火墙没有阻断 80 端口
   服务器上有服务监听请求
   浏览器可以通过域名访问服务
   如果这一步失败，排查方向一般是：
   DNS 是否生效
   A 记录是否配置正确
   服务器 IP 是否正确
   80 端口是否开放
   Nginx 是否启动
   服务器防火墙是否拦截
   云厂商安全组是否放行

10. 一开始不用买哪些东西？
    这次配置过程中，我也确认了一些不必一开始购买的增值服务。
    10.1 付费 DNS 暂时不需要
    个人练习项目，域名注册商提供的基础 DNS 就够用了。
    10.2 付费 SSL 证书暂时不需要
    后续需要 HTTPS，可以使用 Let’s Encrypt 免费证书。
    10.3 备份和安全加固可以后置
    生产环境需要考虑备份、安全防护、监控告警。
    但当前阶段是个人学习项目，优先级应该是：
    先跑通
    再补强
    不要第一天就把预算花在不影响主链路的服务上。

11. 当前阶段结果
    到这里，第一阶段完成。
    当前状态：
    已购买域名
    已购买香港轻量应用服务器
    DNS A 记录解析到服务器公网 IP
    服务器安装 Nginx
    浏览器访问域名可以看到 Nginx 默认页
    链路可以表示为：
    Browser
    ↓
    Domain
    ↓
    DNS
    ↓
    Server Public IP
    ↓
    Nginx
    这一步之后，项目已经拥有了一个真实公网入口。
    后续所有工作都会基于这个入口继续往下走。

12. 后续计划
    后续会继续做：
    关闭宿主机 Nginx
    安装 K3S
    创建 Nginx Demo Deployment
    配置 Ingress Host 为自己的域名
    验证域名 → K3S Ingress → Service → Pod 链路
    构建 Nuxt3 Docker 镜像
    推送镜像到 GHCR
    K3S 部署 Nuxt3 应用
    接入 GitHub Actions 自动部署
    接入 ConfigMap / Secret
    接入 CLS 日志采集
    也就是把链路从：
    域名 → 宿主机 Nginx
    逐步升级为：
    域名 → K3S Ingress → Service → Pod → Nuxt3 应用

13. 总结
    这一篇没有涉及复杂代码。
    但它完成了项目工程化的第一步：
    让一个域名真正访问到自己的服务器。
    对前端来说，这一步很值得做。
    因为它会让你从“本地页面能跑”往后多走一步，开始理解：
    域名怎么解析
    服务器怎么暴露
    端口怎么接收请求
    Web 服务怎么响应
    后续应用如何部署上去
    这也是我做这个 AI Agent 控制台的起点。
    不是先写页面。
    而是先搭一个真实、低成本、能继续扩展的云环境。
