标题
33 岁前端转 AI Agent 工程师，我的第一步不是写代码，而是先买服务器
正文
决定做这个 AI Agent 控制台之后，我没有第一时间写代码。
如果按照以前做前端项目的习惯，我大概率会先建项目、拆页面、写组件、调接口。页面跑起来了，项目看起来也就开始了。
但这一次，我想换一个顺序。
我先做的事情是：买域名，买服务器，让浏览器可以通过一个真实域名访问到我的服务器。
这件事不复杂，甚至有点基础。
但对我来说，它很重要。
因为过去做前端项目，很多基础设施其实都是别人准备好的。服务器、域名、网关、部署环境、日志系统、K8s 集群，通常已经存在。前端工程师更多是在一个已经搭好的系统里，完成页面、交互和接口联调。
而我现在想练的不是某一个页面，也不是某一个组件。
我想练的是：
从一个想法，到一个可以在线访问的产品，它中间到底要经过哪些环节。
所以这次项目的第一步，不是写页面，而是先搭一个低成本、能试错、后面还能继续扩展的云环境。

为什么先从云环境开始？
我做这个项目的背景，是想从传统前端逐步转向 AI Agent 工程方向。
公司里我参与过超级智能体项目，主要负责前端部分，比如：

- AI 流式输出；
- SSE 协议解析；
- 工具调用状态展示；
- 对话页交互；
- 生成中和生成完成后的状态处理。
  这些经验让我对 AI Agent 的前端展示层比较熟悉。
  但我也清楚，自己还缺几块能力：
- 服务端怎么承载 Agent 流程；
- 模型调用怎么封装；
- 工具调用怎么设计；
- Docker 镜像怎么构建；
- 应用怎么部署到容器环境；
- 日志怎么采集；
- 线上出问题怎么排查。
  如果我只是继续写一个前端页面，这些短板不会被补上。
  所以这次我决定先从最外层开始。
  先把域名、服务器、公网访问链路跑通，再往里面放 Nginx、K3S、Ingress、Nuxt 应用、CI/CD 和日志采集。
  这样做的好处是：
  我不是只做一个本地 Demo，而是从第一天就让它朝着“真实可部署产品”的方向走。

我的选型原则
这次选型没有追求完美。
我给自己的原则很简单：

不用备案
成本低
能练 Docker / K3S / Ingress
后面可以继续扩展
不要一开始就把预算打满

因为这个项目最初就是学习和试错，我不想把大量时间花在备案、复杂云产品和高成本资源上。
最后我选的是：

腾讯云香港轻量应用服务器
一个低成本 .top 域名

这不是最强方案，但对我当前阶段来说足够合适。

服务器为什么选香港轻量应用服务器？
服务器我选的是腾讯云香港轻量应用服务器。
配置大概是：

2 核 CPU
2G 内存
40GB SSD
每月 0.5TB 流量
Ubuntu 22.04
中国香港
38 元 / 月

选香港，主要是为了免备案。
如果选择中国内地服务器，域名要正式访问，一般需要备案。对一个练习项目来说，这会增加不少等待时间。
而我现在更关心的是：
今天买，今天能不能跑起来。
所以香港服务器更适合这个阶段。
配置上，我没有选 1 核 2G，而是选了 2 核 2G。
原因是后面计划跑 K3S。
1 核不是完全不能跑，但余量太小。后面如果同时跑 Nuxt 应用、K3S、Ingress、日志采集，一旦资源紧张，排查问题会很烦。
我不想刚开始练云原生，就把时间消耗在机器资源不够导致的各种问题上。
这个配置跑 Nginx、K3S、Ingress、Nuxt 应用和一些测试服务，前期基本够用。
不过这里也踩了一个认知点。
轻量应用服务器虽然便宜好用，但它不是 CVM。
这会影响后面使用腾讯云 TKE。
我最开始想的是：镜像推到腾讯云镜像仓库，然后用腾讯云 TKE 部署。
后来发现，我买的是香港轻量应用服务器，不是 CVM，这条路不能按原计划直接走。
所以后面方案从：

TCR + TKE 托管 Kubernetes

调整成了：

GitHub Container Registry + 自建单节点 K3S

这算是这次选型里的一个小缺点。
但也正因为这个限制，我反而补了一次更完整的自建 K8s 练习。

域名为什么选 .top？
域名这块，我一开始也纠结过要不要买 .com。
.com 确实更正式，但对一个练习项目来说有点没必要。
我最后选了 .top。
一年十几块钱，成本很低，解析到香港服务器也不用备案。前期做项目练习、环境测试和功能调试，完全够用。
如果项目后面真的做起来，再买一个正式一点的 .com 域名也来得及。
到时候只需要把新域名解析到现有服务器，整体架构不用推倒重来。
这里也顺便记录一个小知识点：
买了一个顶级域名之后，二级域名不需要额外付费。
比如：

api.xxx.top
admin.xxx.top
demo.xxx.top

这些都可以在 DNS 控制台里免费添加解析记录。
这对后面做服务拆分会很方便。

第一条链路：让域名真正访问到服务器
服务器和域名买好之后，我做的第一件事是配置 DNS。
核心就是添加一条 A 记录：

域名 -> 服务器公网 IP

解析生效后，浏览器就可以通过域名找到这台服务器。
但这还不够。
服务器上还需要有服务接住这个请求。
所以我先安装了 Nginx。
这一步没有复杂配置，目的只有一个：
验证这条链路是不是通的：

浏览器
→ 域名 DNS
→ 服务器公网 IP
→ 服务器 80 端口
→ Nginx

安装并启动 Nginx 后，我在浏览器里访问自己的域名，看到了 Nginx 默认欢迎页。
这个页面本身没什么价值。
但对我来说，它代表第一条公网访问链路打通了。
以前说“页面上线”，我更多关注的是前端代码有没有构建成功。
这次自己从域名和服务器开始搭，感受会更具体：
一个页面能被访问到，背后不是只有代码。
DNS、服务器、公网 IP、端口、Web 服务，这些都要正常工作。
任何一个环节不通，浏览器里看到的可能就是：

域名解析失败
连接超时
拒绝连接
502
404

这些东西以前可能是运维或者后端同事处理。
但如果想成为一个具备完整交付能力的开发者，就不能永远只停留在浏览器这一侧。

有些东西，一开始不用急着买
这次下单和配置过程中，我也确认了几个容易让人多花钱的点。
第一，付费 DNS 暂时没必要。
域名注册后，一般会提供基础 DNS 解析能力。个人练习项目完全够用。
第二，付费 SSL 证书暂时没必要。
后面需要 HTTPS 的时候，可以用 Let’s Encrypt 免费证书。个人项目没必要一开始就买付费证书。
第三，服务器附带的备份、安全加固、付费监控，可以先不选。
这些东西对生产环境当然有价值，但我当前只是学习和验证，先控制成本更重要。
这也是我这次项目的一个原则：
先跑通，再补强。
不要一开始就把自己困在一堆“看起来很专业”的选项里。

这一阶段完成了什么？
第一阶段结束后，我完成的事情其实很简单：

买好了域名
买好了香港轻量应用服务器
DNS 解析到了服务器公网 IP
服务器上安装了 Nginx
浏览器访问域名，可以看到 Nginx 默认页面

没有写业务代码。
没有接 AI。
没有 Agent。
但我确认了一件事：
我已经有了一个自己的公网入口。
后面无论是部署 Nuxt 应用，还是安装 K3S、配置 Ingress、接入 CI/CD 和日志采集，本质上都要从这个入口继续往里搭。

给前端同学的一点建议
如果你也是前端，也想补齐一点部署和云原生能力，我现在会建议：
第一，不要一开始就追求完美架构。
练习项目最重要的是先跑起来，而不是第一天就设计出生产级系统。
第二，低成本云服务器很适合试错。
香港轻量服务器免备案、成本低、上手快，对个人练习很友好。但也要知道，它和 CVM 不完全一样，后面接托管 K8s 时可能会有限制。
第三，域名不用一开始就买贵的。
.top、.online 这类低成本域名，完全可以支撑前期学习和测试。
第四，少买增值服务。
备份、安全加固、付费监控、付费证书这些，不是不重要，而是可以后置。
第五，先落地，再优化。
先让域名能访问服务器，再让服务器跑应用，再让应用进入容器，最后再考虑 K3S、CI/CD 和日志采集。

最后
这是我重新部署自己的第一步。
不是先写一个炫酷页面。
而是先用低成本搭出一个真实入口，让一个域名真正访问到我自己的服务器。
这一步技术难度不算高，但对我有意义。
因为它让我从“只关心页面怎么显示”，开始往后多走几步，去理解：

请求从哪里来
域名怎么解析
服务器怎么接住
服务怎么暴露
应用最终怎么部署

下一篇，我会记录第一次计划变化：
我原本以为可以直接使用腾讯云 TKE，但后来发现轻量应用服务器不支持这条路。
于是部署方案从托管 Kubernetes，改成了自己在服务器上安装单节点 K3S。
