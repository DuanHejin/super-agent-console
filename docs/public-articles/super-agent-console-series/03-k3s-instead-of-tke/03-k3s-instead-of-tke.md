# 计划赶不上现实：从 TKE 改成自己安装 K3S

第一篇里，我先买了域名和香港轻量应用服务器。

那一步的目标很简单：先让一个域名真正访问到我自己的服务器。

当时我还没有开始写业务代码，只是先把最外层的公网入口打通。

接下来，我原本准备进入“云原生部署”这一层。

一开始的想法其实挺直接：

既然公司里的生产环境是 Kubernetes，那我也尽量按照类似的方式来练。

我想把这个项目部署到腾讯云 TKE 托管 Kubernetes 集群里，再逐步接入 Nacos、CLS 日志采集这些能力。

这样做的好处很明显。

我可以少操心集群本身的维护，把更多精力放在应用部署、配置管理、日志采集和 CI/CD 上。

对一个前端来说，能直接基于云厂商托管服务练习 Kubernetes，也更接近真实生产环境。

所以我最初的设想是：

```txt
Nuxt 应用
→ Docker 镜像
→ 镜像仓库
→ 腾讯云 TKE
→ Ingress 暴露服务
→ Nacos 管理配置
→ CLS 采集日志
```

听起来挺完整。

但很快，现实给了我一个小提醒：

我买的是轻量应用服务器，不是 CVM。

而腾讯云 TKE 托管集群，并不能直接把轻量应用服务器作为节点接进去。

TKE 需要的是标准 CVM 节点。

这就意味着，如果继续走 TKE，我需要重新购买 CVM，甚至还要考虑负载均衡、节点规格、网络等资源。

对公司项目来说，这很正常。

但对我这个个人练习项目来说，成本就开始偏高了。

我一开始选香港轻量服务器，就是为了低成本、免备案、快速试错。

如果为了练 Kubernetes，又重新买一套 CVM 资源，那就偏离了这个项目的初衷。

这时候就出现了第一个现实选择：

是继续追求更像生产环境的 TKE，还是先用已有服务器把 Kubernetes 这条路跑通？

我最后选择了后者。

也就是：不再使用 TKE，而是在这台香港轻量应用服务器上自己安装单节点 K3S。

## 为什么是 K3S

K3S 可以理解为一个更轻量的 Kubernetes 发行版。

它适合资源有限的环境，也适合边缘节点、开发测试、小型集群。

对我当前这台 2 核 2G 的轻量服务器来说，K3S 比完整自建一套标准 Kubernetes 更合适。

它的优势很明确：

```txt
安装简单
资源占用更低
保留 Kubernetes 的核心能力
支持 Deployment / Service / Ingress
适合个人练习和小项目部署
```

我不是要用它承载复杂生产系统。

我只是希望通过它理解：

```txt
一个容器应用如何部署到 Kubernetes
Service 如何暴露容器
Ingress 如何接入域名流量
Pod 如何滚动更新
后续 CI/CD 如何修改镜像版本
```

这些能力，K3S 都能练到。

所以，虽然它不是公司生产环境里的 TKE，但它足够让我继续往前走。

这也是这次项目给我的一个提醒：

学习阶段不一定要完全复刻生产环境。

有时候，能用低成本跑通核心链路，比一开始追求一套“看起来更正规”的方案更重要。

## 关闭宿主机 Nginx

第一篇里，我在服务器上安装了 Nginx，用来验证域名到服务器的公网链路。

但如果后面要让 K3S 的 Ingress 接管 80 端口，宿主机 Nginx 就不能继续占着端口。

所以在安装 K3S 前，我先把宿主机上的 Nginx 停掉。

这一步的意义是：

```txt
之前：浏览器 → 域名 → 服务器 Nginx
之后：浏览器 → 域名 → K3S Ingress → 应用服务
```

也就是说，入口要从宿主机 Nginx 切换到 Kubernetes 里的 Ingress。

这是从“服务器上跑一个 Web 服务”，进入到“服务器上跑一个容器编排环境”的关键变化。

## 安装 K3S

K3S 的安装过程相对简单。

安装完成后，会在服务器上得到一个可以使用的 Kubernetes 环境。

这时候就可以使用 `kubectl` 查看节点和 Pod 状态。

这一步跑通后，我其实已经完成了一个很重要的切换：

服务器不再只是一个安装了 Nginx 的 Linux 主机。

它开始变成一个小型 Kubernetes 节点。

虽然只是单节点，但 Deployment、Service、Ingress 这些概念都可以真实运行起来。

对我来说，这比单纯看 Kubernetes 文档要有效得多。

以前我知道这些概念，但更多停留在“听过、用过别人配好的环境”。

这次自己装完 K3S 后，才更清楚地意识到：

Kubernetes 不是一个抽象名词。

它最终还是要落在某台机器上，接管容器、网络、端口和流量。

## 用 Nginx Demo 验证 K3S 链路

安装完 K3S 后，我没有马上部署自己的 Nuxt 项目。

而是先写了一个最简单的 demo 应用。

这个 demo 的 Deployment 里，容器镜像用的是官方 Nginx。

Service 用来暴露这个 Nginx Pod。

Ingress 的 host 改成我自己的域名。

这样做的目的很简单：

先验证 K3S 这条新链路是不是通的。

最终链路变成：

```txt
浏览器
→ 域名
→ 服务器公网 IP
→ K3S Ingress
→ Service
→ Nginx Pod
```

再次打开浏览器访问域名，如果还能看到 Nginx 默认页，就说明这一次页面不是由宿主机 Nginx 返回的，而是由 K3S 里的 Nginx Pod 返回的。

这一步让我很踏实。

因为它说明：

```txt
域名入口还在
服务器公网链路还在
K3S Ingress 能接到请求
Service 能转发到 Pod
Pod 能正常响应
```

到这里，K3S 的基础部署链路就跑通了。

## 这次调整带来的变化

从 TKE 改成 K3S，并不是我一开始计划好的。

它更像是项目推进过程中遇到现实限制后的调整。

一开始我确实希望尽量靠近公司的云原生架构：

```txt
K8s
Nacos
CLS
托管服务
自动化部署
```

但个人项目和公司项目不一样。

公司项目优先考虑稳定性、团队协作、权限管理、生产保障。

个人练习项目优先考虑的是：

```txt
成本
可控
能不能快速试错
能不能真正理解链路
```

TKE 更标准，但对我的轻量服务器不适配。

K3S 没有托管服务那么省心，但它让我更直接地接触 Kubernetes 本身。

这其实也挺符合我当前的状态。

我不是在搭一个企业生产集群，而是在重新补齐自己对部署和云原生的理解。

所以这次调整没有让我觉得项目失败，反而让我对“工程选择”有了更具体的感受：

技术方案不是越完整越好，而是要和当前目标匹配。

## 当前结果

第二阶段完成后，我得到的结果是：

```txt
确认 TKE 不支持轻量应用服务器作为节点
放弃原来的 TKE 托管方案
在香港轻量服务器上安装 K3S
关闭宿主机 Nginx，让 K3S Ingress 接管入口
部署 Nginx Demo 到 K3S
通过自己的域名访问到 K3S 内的 Nginx Pod
```

也就是说，公网流量已经从：

```txt
域名 → 宿主机 Nginx
```

切换成了：

```txt
域名 → K3S Ingress → Service → Pod
```

这是一个很重要的变化。

它意味着后面我的 Nuxt 应用不再只是跑在服务器上，而是可以作为一个容器应用部署到 K3S 里。

## 给前端和初学者的一点建议

如果你也是个人练习项目，我现在会建议：

第一，不要默认云厂商所有产品都能互相接上。

轻量应用服务器和 CVM 是不同产品。很多托管能力，比如 TKE，通常依赖标准 CVM 节点。

第二，先明确目标是练习还是生产。

如果是生产环境，托管 Kubernetes 更省心，也更适合团队使用。

但如果是个人学习，单节点 K3S 更便宜，也更容易理解底层链路。

第三，不要一开始就追求完全复刻公司架构。

公司架构背后有团队、预算和运维体系。个人练习更重要的是先跑通主链路。

第四，先用 Nginx Demo 验证链路。

不要一上来就部署自己的复杂项目。先用最简单的镜像确认 Ingress、Service、Pod 是通的，排错成本会低很多。

第五，方案变化不是坏事。

这次从 TKE 改成 K3S，就是一个典型例子。

一开始的计划看起来更接近生产，但现实条件不匹配。调整之后，虽然方案变轻了，但学习价值并没有降低。

这就是我重新部署自己的第二步。

我原本想直接站在托管 Kubernetes 上练习。

最后却选择从一台轻量服务器上的单节点 K3S 开始。

它不完美，但刚好适合现在的我。
