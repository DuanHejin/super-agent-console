标题
计划赶不上现实：我原本想用 TKE，最后却自己装了 K3S
正文
上一篇里，我记录了自己做 AI Agent 控制台的第一步：买域名、买服务器，让一个域名真正访问到自己的云服务器。
那一步没有写业务代码，也没有接 AI，只是先把最外层的公网入口打通。
接下来，我原本准备进入“云原生部署”这一层。
一开始的想法其实挺直接：
既然公司里的生产环境是 Kubernetes，那我个人项目也尽量按照类似方式来练。
我想把这个项目部署到腾讯云 TKE 托管 Kubernetes 集群里，然后再逐步接入配置管理、日志采集、CI/CD 这些能力。
当时设想的链路大概是：

Nuxt 应用
→ Docker 镜像
→ 镜像仓库
→ 腾讯云 TKE
→ Ingress 暴露服务
→ 配置管理
→ CLS 日志采集

这个方案看起来很完整，也更接近真实生产环境。
但很快，现实给了我一个提醒：
我买的是轻量应用服务器，不是 CVM。
而腾讯云 TKE 不能直接把轻量应用服务器作为节点接进去。TKE 需要的是标准 CVM 节点。
这就意味着，如果继续走 TKE，我需要重新购买 CVM，还要考虑节点规格、网络、负载均衡等资源。
对公司项目来说，这很正常。
但对我这个个人练习项目来说，成本和复杂度就开始偏高了。
我一开始选择香港轻量应用服务器，就是为了：

低成本
免备案
快速试错
先把主链路跑通

如果为了练 Kubernetes，又重新买一套 CVM 资源，就偏离了这个项目的初衷。
于是，这里出现了第一个现实选择：
是继续追求更接近生产环境的 TKE，还是先基于已有服务器把 Kubernetes 这条链路跑通？
我最后选择了后者。
也就是：放弃 TKE，在这台香港轻量应用服务器上自己安装单节点 K3S。

为什么改用 K3S？
K3S 可以理解为一个更轻量的 Kubernetes 发行版。
它适合资源有限的环境，也适合边缘节点、开发测试和小型集群。
对我当前这台 2 核 2G 的轻量服务器来说，K3S 比自己搭一套完整 Kubernetes 更合适。
它的优势很明确：

安装简单
资源占用较低
保留 Kubernetes 核心能力
支持 Deployment / Service / Ingress
适合个人练习和小项目部署

我不是要用它承载复杂生产系统。
我只是希望通过它理解：

容器应用如何部署到 Kubernetes
Deployment 如何管理 Pod
Service 如何暴露应用
Ingress 如何接入域名流量
Pod 如何被更新
后续 CI/CD 如何修改镜像版本

这些能力，K3S 都能练到。
所以，虽然它不是公司生产环境里的 TKE，但对我当前阶段来说已经足够。
这也是这次项目给我的一个提醒：
学习阶段不一定要完全复刻生产环境。 能用低成本跑通核心链路，比一开始追求“看起来更正规”的方案更重要。

关闭宿主机 Nginx
上一篇里，我在服务器上安装了 Nginx，用它验证域名到服务器的公网链路。
当时访问路径是：

浏览器
→ 域名
→ 服务器公网 IP
→ 宿主机 Nginx

但现在要让 K3S 的 Ingress 接管入口，宿主机 Nginx 就不能继续占用 80 端口。
所以在安装 K3S 前，我先把宿主机上的 Nginx 停掉。
这一步的意义是：

之前：浏览器 → 域名 → 宿主机 Nginx
之后：浏览器 → 域名 → K3S Ingress → 应用服务

入口从“服务器上直接跑一个 Web 服务”，切换到了“Kubernetes 里的 Ingress”。
这一步看似简单，但它标志着项目进入了另一个阶段：
从普通服务器部署，进入容器编排环境。

安装 K3S 后的感觉
K3S 安装过程相对简单。
安装完成后，我就可以用 kubectl 查看节点和 Pod 状态。
这一步跑通之后，我其实完成了一个很重要的切换：
服务器不再只是一个安装了 Nginx 的 Linux 主机。
它开始变成了一个小型 Kubernetes 节点。
虽然只是单节点，但 Deployment、Service、Ingress 这些概念都能真实运行起来。
以前我对 Kubernetes 的很多理解，更多来自公司现成环境。
我知道 Pod 是什么，知道 Deployment 是什么，也查过容器日志，但很多时候是在别人已经搭好的体系里使用。
这次自己安装 K3S 后，感受会更具体：
Kubernetes 不是一个抽象名词。 它最终还是要落在某台机器上，接管容器、网络、端口和流量。
这比单纯看文档有效很多。

用 Nginx Demo 验证 K3S 链路
安装完 K3S 后，我没有马上部署自己的 Nuxt 项目。
而是先写了一个最简单的 demo。
这个 demo 很简单：

Deployment 使用官方 Nginx 镜像
Service 暴露 Nginx Pod
Ingress 的 host 配成自己的域名

这样做的目的只有一个：
先验证 K3S 这条新链路是不是通的。
最终链路变成：

浏览器
→ 域名
→ 服务器公网 IP
→ K3S Ingress
→ Service
→ Nginx Pod

再次打开浏览器访问域名，如果还能看到 Nginx 默认页，就说明这次页面不是由宿主机 Nginx 返回的，而是由 K3S 里的 Nginx Pod 返回的。
这一步让我很踏实。
因为它说明几件事都正常了：

域名入口仍然可用
服务器公网链路正常
K3S Ingress 能接到请求
Service 能转发到 Pod
Pod 能正常响应

到这里，K3S 的基础部署链路就跑通了。

从 TKE 改成 K3S，算不算退而求其次？
表面上看，好像是。
TKE 是云厂商托管服务，更标准、更省心，也更接近公司生产环境。
K3S 是自己在轻量服务器上搭的单节点集群，看起来没那么“企业级”。
但从我当前目标看，这并不是退步。
因为我现在不是在搭企业生产集群，而是在补齐自己的工程链路理解。
公司项目和个人项目不一样。
公司项目更关注：

稳定性
团队协作
权限管理
生产保障
运维体系

个人练习项目更关注：

成本
可控
快速试错
能不能理解主链路
能不能真正跑起来

TKE 更标准，但对我的轻量服务器不适配。
K3S 没有托管服务那么省心，但它让我更直接地接触 Kubernetes 本身。
这其实也挺适合我当前阶段。
我不是为了证明自己会搭一套多节点生产集群，而是为了理解：

镜像如何变成 Pod
Service 如何转发流量
Ingress 如何接入域名
Deployment 如何更新容器

这些核心概念，在 K3S 里都能练到。

当前阶段完成了什么？
第二阶段完成后，我得到的结果是：

确认 TKE 不支持轻量应用服务器作为节点
放弃原来的 TKE 托管方案
在香港轻量服务器上安装 K3S
关闭宿主机 Nginx，让 K3S Ingress 接管入口
部署 Nginx Demo 到 K3S
通过自己的域名访问到 K3S 内的 Nginx Pod

公网流量已经从：

域名 → 宿主机 Nginx

切换成了：

域名 → K3S Ingress → Service → Pod

这是一个很重要的变化。
它意味着后面我的 Nuxt 应用不再只是跑在服务器上，而是可以作为一个容器应用部署到 K3S 里。

给前端和初学者的一点建议
如果你也是个人练习项目，我现在会建议：
第一，不要默认云厂商所有产品都能直接接上。
轻量应用服务器和 CVM 是不同产品。很多托管能力，比如 TKE，通常依赖标准 CVM 节点。
第二，先明确自己是在学习，还是在做生产系统。
如果是生产系统，托管 Kubernetes 更省心，也更适合团队使用。 但如果是个人学习，单节点 K3S 更便宜，也更容易理解底层链路。
第三，不要一开始就完全复刻公司架构。
公司架构背后有团队、预算和运维体系。个人练习更重要的是先跑通主链路。
第四，先用 Nginx Demo 验证链路。
不要一上来就部署自己的复杂项目。先用最简单的镜像确认 Ingress、Service、Pod 是通的，排错成本会低很多。
第五，方案变化不是坏事。
这次从 TKE 改成 K3S，就是一个典型例子。
一开始的计划看起来更接近生产，但现实条件不匹配。调整之后，虽然方案变轻了，但学习价值并没有降低。

最后
这是我重新部署自己的第二步。
我原本想直接站在托管 Kubernetes 上练习，最后却选择从一台轻量服务器上的单节点 K3S 开始。
它不完美，但刚好适合现在的我。
这次调整也让我意识到：
工程方案不是越完整越好，而是要和当前阶段、预算、目标匹配。
