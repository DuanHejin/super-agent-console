## 我以为线上部署完成了，结果卡在服务器到大模型的网络链路

到这里，我一度以为线上部署差不多完成了。

域名能访问。

HTTPS 正常。

登录正常。

Cookie 正常。

数据库正常。

反馈能提交。

管理员页面能看。

CLS 日志也能采集。

从传统 Web 项目的角度看，这套链路已经基本闭环了。

但真正开始测试 Agent Run 时，新的问题出现了。

线上经常失败。

我去 CLS 里看日志，发现有一条 Run 是这样的：

```txt
model_call_start
↓
大约 60 秒
↓
agent_error
```

也就是说，Agent 已经进入真实模型调用阶段，但还没等到下一步结果，就失败了。

一开始我怀疑是代码问题。

是不是 tool call 解析有问题？

是不是 Skill 调模型出错？

是不是模型返回格式不符合 JSON？

但继续看日志发现，这次失败发生在第一次模型调用阶段。

也就是 tool planning。

这个阶段模型还没返回 tool call，Tool 和 Skill 都还没开始执行。

再看时间，刚好卡在 60 秒左右。

而我的配置里：

```txt
MODEL_REQUEST_TIMEOUT_MS=60000
```

基本可以判断，是模型请求超时了。

于是我先做了两个调整。

第一个，是把超时时间从 60 秒调到 180 秒。

```txt
MODEL_REQUEST_TIMEOUT_MS=180000
```

第二个，是增强 `agent_error` 日志。

之前 CLS 里只能看到：

```txt
Real Agent Run failed
```

这个信息太粗了。

看不出来到底失败在哪个阶段，也看不出来是不是超时。

所以我给错误日志加了几个字段：

```txt
phase
errorMessage
errorName
isTimeout
requestTimeoutMs
```

这样以后再出问题，就能知道：

```txt
是 tool_planning 阶段失败
还是 tool_execution 阶段失败
还是 final_answer 阶段失败
是不是超时
当时超时时间是多少
```

但这只是让问题更容易观察。

它没有真正解释，为什么线上会慢。

后来我直接在香港服务器上 curl 豆包的 Chat API。

结果很明显：

有时候很久才返回。

有时候失败。

有时候第三次才成功，而且花了三十多秒。

但我在本地测试，相对正常。

这让我开始意识到，问题可能不在代码。

而在链路。

我的服务器在中国香港。

豆包 Seed 2.0 Lite 接在火山方舟北京节点。

也就是说，线上模型请求是：

```txt
香港轻量服务器
↓
跨境网络
↓
火山方舟北京接口
```

AI 应用和普通网站有一点不一样。

普通网站里，服务器访问数据库、对象存储、缓存，一般都在同地域或同云内网里。

但我这个项目里，最关键的一段链路是服务器访问大模型平台。

如果这条链路不稳定，页面做得再好，Agent Runtime 写得再完整，最终用户看到的还是失败。

这件事让我有点无奈。

因为前面一直在补代码能力：

```txt
SSE
AgentEvent
Tool Router
Skill Workflow
Prisma
CLS
登录
频控
HTTPS
```

但最后卡住的，却是服务器和模型之间的网络距离。

更现实的问题是：

如果我把服务器迁到中国大陆，访问火山方舟北京大概率会稳定很多。

但新的问题又来了。

我现在的镜像仓库用的是 GHCR。

GitHub Container Registry 在国外。

大陆服务器从 GHCR 拉镜像，可能也会慢，甚至不稳定。

也就是说：

```txt
香港服务器：
访问 GHCR 可能还行
访问国内模型不稳定

大陆服务器：
访问国内模型更稳定
访问 GHCR 可能不稳定
```

这就是部署链路里的取舍。

所以我整理了一个备选方案：

```txt
国内轻量服务器
腾讯云 TCR 镜像仓库
自建 K3S
火山方舟北京模型
```

TCR 是腾讯云的容器镜像服务。

它是独立的镜像仓库，不绑定 TKE。

虽然腾讯云 TKE 不支持轻量应用服务器作为节点，但这不影响我在轻量服务器上自建 K3S，然后从 TCR 拉镜像。

新的链路会变成：

```txt
GitHub Actions 构建镜像
↓
推送到腾讯云 TCR
↓
国内 K3S 从 TCR 拉镜像
↓
国内服务器访问火山方舟北京接口
```

这样至少两个关键链路都在国内网络里：

```txt
K3S → TCR
K3S → 火山方舟
```

当然，迁到国内服务器也不是没有代价。

第一个代价是备案。

香港服务器可以免备案，域名解析后很快就能访问。

但大陆服务器正式绑定域名，需要 ICP 备案。

备案不只是等几天，还涉及网站名称、个人主体、内容说明、材料核验等。

第二个代价是带宽。

我看了南京轻量应用服务器，2 核 2G、40G SSD、2Mbps 带宽、100GB 流量，大概 35 元一个月。

价格和香港差不多。

但香港是 20Mbps 峰值带宽、512GB 流量。

从带宽看，南京弱很多。

不过对我的 Agent 项目来说，核心瓶颈不是大文件下载，而是服务器到模型服务的稳定性。

SSE 文本流本身不太吃带宽。

所以如果只是少量朋友内测，南京 2Mbps 也不是不能用。

第三个代价是迁移。

需要重新搭：

```txt
K3S
MySQL
ConfigMap
Secret
Ingress
HTTPS
CLS LogListener
GitHub Actions SSH 目标
TCR imagePullSecret
```

数据库也不能直接复制 PVC，最好用 mysqldump 导出，再导入新服务器。

所以我没有马上迁。

而是先把迁移步骤整理成文档，作为备选方案。

如果香港服务器通过调大超时、降低 max tokens、优化模型请求之后还能撑住内测，就先继续用香港。

如果还是频繁超时，再切国内服务器。

这一阶段让我对“上线”这件事有了更现实的认识。

以前我以为，部署完成就是：

```txt
域名能打开
服务能访问
日志能采集
数据库能写入
```

但 AI 应用还多了一层：

```txt
服务器能不能稳定访问模型
模型首包会不会太慢
流式响应会不会中断
上游接口和部署地域是否匹配
```

也就是说，AI 应用的稳定性，不只取决于我的代码。

还取决于服务器、模型平台、镜像仓库、网络地域这些基础设施选择。

这也是这几天最真实的感受：

项目不是没上线。

而是上线之后才发现，真正的线上环境，会把所有“之前没认真考虑的问题”都推到面前。

代码只是其中一部分。

剩下的，是工程。
