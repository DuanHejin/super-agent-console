标题

我以为线上部署完成了，结果卡在服务器到大模型的网络链路

正文

到这里，我一度以为线上部署差不多完成了。

域名能访问。

HTTPS 正常。

登录正常。

Cookie 正常。

数据库正常。

反馈能提交。

管理员页面能看。

CLS 日志也能采集。

从传统 Web 项目的角度看，这套链路已经基本闭环了。

但真正开始测试 Agent Run 时，新的问题出现了。

线上经常失败。

我去 CLS 里看日志，发现有一条 Run 是这样的：

```txt
model_call_start
↓
大约 60 秒
↓
agent_error
```

也就是说，Agent 已经进入真实模型调用阶段，但还没等到下一步结果，就失败了。

一开始我怀疑是代码问题。

是不是 tool call 解析有问题？

是不是 Skill 调模型出错？

是不是模型返回格式不符合 JSON？

但继续看日志发现，这次失败发生在第一次模型调用阶段。

也就是 tool planning。

这个阶段模型还没返回 tool call，Tool 和 Skill 都还没开始执行。

再看时间，刚好卡在 60 秒左右。

而我的配置里正好有：

```txt
MODEL_REQUEST_TIMEOUT_MS=60000
```

基本可以判断：

这次不是 Tool 错了。

也不是 Skill 错了。

而是模型请求超时了。

## 先把错误变得更可观察

我先做了两个调整。

第一个，是把模型请求超时时间从 60 秒调到 180 秒。

```txt
MODEL_REQUEST_TIMEOUT_MS=180000
```

这不是最终解决方案。

但至少可以先验证：是不是模型首包太慢，导致 60 秒内等不到响应。

第二个，是增强 `agent_error` 日志。

之前 CLS 里只能看到：

```txt
Real Agent Run failed
```

这个信息太粗了。

看不出来到底失败在哪个阶段。

也看不出来是不是超时。

所以我给错误日志加了几个字段：

```txt
phase
errorMessage
errorName
isTimeout
requestTimeoutMs
```

这样以后再出问题，就能知道：

```txt
是 tool_planning 阶段失败
还是 tool_execution 阶段失败
还是 final_answer 阶段失败
是不是超时
当时超时时间是多少
```

这一步只是让问题更容易观察。

它没有真正解决问题。

但在排查线上问题时，能不能先看清楚问题，已经很重要。

## 问题可能不在代码，而在链路

后来我直接在香港服务器上 curl 豆包的 Chat API。

结果很明显：

有时候很久才返回。

有时候失败。

有时候第三次才成功，而且花了三十多秒。

但我在本地测试，相对正常。

这让我开始意识到：

问题可能不在代码。

而在链路。

我的服务器在中国香港。

豆包 Seed 2.0 Lite 接在火山方舟北京节点。

也就是说，线上模型请求是：

```txt
香港轻量服务器
↓
跨境网络
↓
火山方舟北京接口
```

这段链路本身就不稳定。

AI 应用和普通网站有一点不一样。

普通网站里，服务器访问数据库、对象存储、缓存，一般都在同地域或同云内网里。

但我这个项目里，最关键的一段链路是服务器访问大模型平台。

如果这条链路不稳定，页面做得再好，Agent Runtime 写得再完整，最终用户看到的还是失败。

这件事让我有点无奈。

因为前面一直在补代码能力：

```txt
SSE
AgentEvent
Tool Router
Skill Workflow
Prisma
CLS
登录
频控
HTTPS
```

但最后卡住的，却是服务器和模型之间的网络距离。

## AI 应用多了一条关键链路

以前我理解线上部署，更多是：

```txt
域名能打开
服务能访问
日志能采集
数据库能写入
```

这些做好之后，传统 Web 项目基本就能跑了。

但 AI 应用多了一条关键链路：

```txt
服务器
↓
大模型平台
```

这条链路如果不稳定，整个 Agent 就不稳定。

尤其是 Agent 项目里，一次 Run 可能不止调用一次模型。

它可能有：

```txt
tool planning 模型调用
Skill 内部模型调用
final answer 模型调用
```

一次 Run 里模型调用越多，对链路稳定性的依赖就越强。

所以模型接口不是普通的第三方接口。

它几乎是 Agent 项目的核心基础设施。

这也是这次线上排查给我的一个新认知。

## 香港服务器和国内模型之间的取舍

更现实的问题是：

如果我把服务器迁到中国大陆，访问火山方舟北京大概率会稳定很多。

但新的问题又来了。

我现在的镜像仓库用的是 GHCR。

GitHub Container Registry 在国外。

大陆服务器从 GHCR 拉镜像，可能也会慢，甚至不稳定。

也就是说：

```txt
香港服务器：
访问 GHCR 可能还行
访问国内模型不稳定

大陆服务器：
访问国内模型更稳定
访问 GHCR 可能不稳定
```

这就是部署链路里的取舍。

以前我以为选香港服务器主要是为了免备案。

现在才发现，服务器地域还会影响整个模型调用链路。

如果项目的核心依赖是国内大模型平台，那服务器放在香港，就可能天然存在跨境网络抖动。

这不是代码优化能完全解决的。

## 备选方案：国内轻量服务器 + TCR + K3S

所以我整理了一个备选方案：

```txt
国内轻量服务器
腾讯云 TCR 镜像仓库
自建 K3S
火山方舟北京模型
```

TCR 是腾讯云的容器镜像服务。

它是独立的镜像仓库，不绑定 TKE。

虽然腾讯云 TKE 不支持轻量应用服务器作为节点，但这不影响我在轻量服务器上自建 K3S，然后从 TCR 拉镜像。

新的链路会变成：

```txt
GitHub Actions 构建镜像
↓
推送到腾讯云 TCR
↓
国内 K3S 从 TCR 拉镜像
↓
国内服务器访问火山方舟北京接口
```

这样至少两个关键链路都在国内网络里：

```txt
K3S → TCR
K3S → 火山方舟
```

这比当前链路更适合国内模型。

但它也不是没有代价。

## 迁到国内服务器的代价

第一个代价是备案。

香港服务器可以免备案。

域名解析后很快就能访问。

但大陆服务器正式绑定域名，需要 ICP 备案。

备案不只是等几天，还涉及网站名称、个人主体、内容说明、材料核验等。

这对一个想快速试错的个人项目来说，是一个额外成本。

第二个代价是带宽。

我看了南京轻量应用服务器。

2 核 2G、40G SSD、2Mbps 带宽、100GB 流量，大概 35 元一个月。

价格和香港差不多。

但香港是 20Mbps 峰值带宽、512GB 流量。

从带宽看，南京弱很多。

不过对我的 Agent 项目来说，核心瓶颈不是大文件下载，而是服务器到模型服务的稳定性。

SSE 文本流本身不太吃带宽。

所以如果只是少量朋友内测，南京 2Mbps 也不是不能用。

第三个代价是迁移。

需要重新搭：

```txt
K3S
MySQL
ConfigMap
Secret
Ingress
HTTPS
CLS LogListener
GitHub Actions SSH 目标
TCR imagePullSecret
```

数据库也不能直接复制 PVC。

最好用 `mysqldump` 导出，再导入新服务器。

这些都不是不能做。

但需要时间。

所以我没有马上迁。

而是先把迁移步骤整理成文档，作为备选方案。

如果香港服务器通过调大超时、降低 max tokens、优化模型请求之后还能撑住内测，就先继续用香港。

如果还是频繁超时，再切国内服务器。

## 上线之后，才会看到真正的链路问题

这一阶段让我对“上线”这件事有了更现实的认识。

以前我以为，部署完成就是：

```txt
域名能打开
服务能访问
日志能采集
数据库能写入
```

但 AI 应用还多了一层：

```txt
服务器能不能稳定访问模型
模型首包会不会太慢
流式响应会不会中断
上游接口和部署地域是否匹配
```

也就是说，AI 应用的稳定性，不只取决于我的代码。

还取决于：

```txt
服务器地域
模型平台地域
镜像仓库地域
网络链路
超时配置
模型首包速度
流式响应稳定性
```

这些基础设施选择，会直接影响最终用户体验。

这也是这几天最真实的感受：

项目不是没上线。

而是上线之后才发现，真正的线上环境，会把所有“之前没认真考虑的问题”都推到面前。

代码只是其中一部分。

剩下的，是工程。

从这个角度看，这次超时并不是坏事。

它让我意识到：

一个 AI Agent 项目，真正上线时要考虑的不只是 Agent Runtime 怎么写。

还要考虑：

```txt
服务部署在哪里
模型服务在哪里
镜像从哪里拉
日志在哪里看
出了问题能不能定位到阶段
超时能不能被识别
失败能不能被复盘
```

这一步没有让我把项目做得更“炫”。

但它让我更接近真实线上系统。
