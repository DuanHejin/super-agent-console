知乎版
标题
我给 Agent 做了一个执行回放页，才发现它真的不是聊天页面
正文
Timeline 做完之后，我很快发现一个问题。
前端页面上虽然能看到 Agent 执行过程，但它更像是“现场观看”。
当一次 Agent Run 正在执行时，我可以看到：

Agent 开始
模型分析
Tool 调用
Skill 执行
最终答案生成

但如果这次 Run 已经结束了，我想回头再看一遍，就没那么方便。
尤其是后面要排查问题时，我不可能只盯着页面实时看。
我需要一个地方，能够根据 runId 把一次执行完整复盘出来。
所以我做了一个 Run 详情页：

/runs/:runId

从实时观看到事后复盘
这个页面的目标很简单：
把一次 Agent Run 发生过的事情全部展示出来。
它会展示：

用户输入
runId
traceId
Run 状态
AgentEvent 列表
Tool 调用过程
Skill 执行过程
最终答案
错误信息

从首页执行完一次 mock run 后，可以通过运行元信息里的入口跳到这个详情页。
这样我就能从“实时看过程”，变成“事后复盘过程”。
这一步做完之后，项目的感觉变得不一样了。
之前 Timeline 更像一个执行中的面板。
Run 详情页则像一个执行记录档案。
它记录了这次 Agent 到底做了什么。

为什么 Run 详情页重要？
因为 Agent 的问题经常不是一句话能说清楚的。
比如最终答案不理想，可能有很多原因：

用户输入不完整
模型分析方向偏了
Tool 参数生成错了
Skill 输入映射错了
Skill 输出结构不合理
Tool Result 回填有问题
最终答案阶段没有用好工具结果

如果没有 Run 详情页，我只能看最终答案。
那我很难判断问题到底在哪里。
但有了 Run 详情页，我能沿着事件顺序往回看。
我可以看到：

模型什么时候开始分析
它什么时候调用 Tool
Tool 参数是什么
Skill 输入是什么
Skill 输出是什么
Tool Result 是什么
最终答案是怎么产生的

这就有了排查依据。
Agent 项目最怕的是黑盒。
因为模型本身已经有不确定性，如果工程链路也不可见，那排查问题就只能靠猜。

它其实是一个轻量版 Trace
这个页面其实是一个轻量版 Trace。
公司里可能会用 Coze Loop 或更完整的观测平台。
这些平台可以记录：

模型调用
Prompt
Tool Call
Skill 执行
输入输出
耗时
错误
评测结果

但我现在不想一上来就接这些平台化工具。
因为 MVP 阶段最重要的是先理解 Trace 的价值。
所以我先自己做一个最小版本：

Run 详情页
AgentEvent 列表
Tool / Skill 过程卡片
最终答案展示

它不复杂。
但它已经能让我解释清楚一次 Agent Run。
后面如果真的要接 Coze Loop，我也知道哪些数据应该上报。
这点很重要。
如果没有自己先做一遍，很容易把 Coze Loop 当成一个“高级日志平台”。
但真正做完 Run 详情页后，我更清楚：
Trace 的本质不是收集日志，而是把一次 Agent 执行过程结构化地还原出来。

从内存过渡到数据库
刚开始 Run 详情页读的是内存 run-store。
也就是说，只要服务没有重启，详情页就能看到这次 Run。
这对于 MVP 很方便。
但它只是过渡方案。
因为内存数据有明显问题：

服务重启就没了
多实例之间不共享
不能长期复盘
不能做统计
不能做 replay

所以我后来又把 Run 详情页改成优先读数据库。
读数据库失败或者查不到时，再 fallback 到内存。
这个过渡方式比较稳。
既不影响当前 SSE 演示链路，又能逐步把数据持久化能力接进来。
这也是我现在做这个项目的一个原则：
先让链路跑通，再逐步把临时方案替换成更接近生产的方案。

runId 和 traceId 的区别
这个阶段我也开始更重视 traceId。
runId 代表一次 Agent 执行。
traceId 更偏向排查链路。
比如：

runId：这一次 Agent Run 是谁
traceId：这一次请求链路怎么追踪

在日志、数据库、前端页面里都带上它之后，我可以把一次执行过程串起来。
比如：

前端看到 runId
Run 详情页看到 traceId
CLS 日志里搜索 traceId
数据库里查 AgentEvent

这些信息就不再是分散的。
以前我可能只会说“线上看日志”。
现在我更清楚地知道，日志要和业务对象关联起来才有价值。
对 Agent 项目来说，这个业务对象就是：

Agent Run

可观察性不是上线后才补
对一个 Agent 项目来说，可观察性不是上线之后才补的东西。
它应该从 MVP 阶段就开始留接口。
因为 Agent 的执行过程本来就比普通接口更复杂。
普通接口可能是：

请求
处理
响应

但 Agent Run 可能是：

用户输入
模型分析
工具调用
Skill 执行
工具结果回填
模型继续生成
最终答案返回

中间每一步都可能出问题。
所以如果一开始没有把这些过程记录下来，后面调试会非常痛苦。
Run 详情页就是我给这个项目加的第一个复盘入口。

我对 Agent Console 的理解变了
做完 Run 详情页后，我对“Agent Console”这个名字有了更明确的感觉。
它不是一个聊天页面。
它更像一个控制台。
用户在这里发起任务。
前端在这里观察任务。
开发者在这里复盘任务。
后续运维也可以从这里排查任务。
如果一个 Agent 系统没有这种复盘能力，那它出问题时会非常痛苦。
因为 AI 的不确定性本来就高。
工程侧能做的，就是尽量让每一步都留下痕迹。
Run 详情页就是这个痕迹的入口。

最后
这一阶段做完后，我开始真正感受到：
Agent Console 的价值，不只是把结果展示出来，而是把一次智能体执行过程完整保存下来。
实时 Timeline 解决的是：

我现在能不能看到 Agent 在做什么

Run 详情页解决的是：

这次执行结束后，我还能不能复盘它做了什么

一个是现场观察。
一个是事后追踪。
这两者加起来，Agent 才不再是黑盒。
下一篇，我会继续记录：
如何进一步把这些 Run、Event、Tool Call、Skill Run 数据落到数据库里，让复盘能力从临时内存变成真正持久化。
