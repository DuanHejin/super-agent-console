# 第五篇：我给 Agent 做了一个执行回放页

Timeline 做完之后，我很快发现一个问题。

前端页面上虽然能看到 Agent 执行过程，但它更像是“现场观看”。

如果这次 Run 已经结束了，我想回头看一遍，就没那么方便。

尤其是后面要排查问题时，我不可能只盯着页面实时看。

我需要一个地方，能够根据 `runId` 把一次执行完整复盘出来。

所以我做了一个 Run 详情页：

```txt
/runs/:runId
```

---

这个页面的目标很简单：

把一次 Agent Run 发生过的事情全部展示出来。

它会展示：

```txt
用户输入
runId
traceId
Run 状态
AgentEvent 列表
Tool 调用过程
Skill 执行过程
最终答案
错误信息
```

从首页执行完一次 mock run 后，可以通过运行元信息里的入口跳到这个详情页。

这样我就能从“实时看过程”，变成“事后复盘过程”。

---

为什么这个页面重要？

因为 Agent 的问题经常不是一句话能说清楚的。

比如最终答案不理想，可能有很多原因：

```txt
用户输入不完整
模型分析方向偏了
Tool 参数生成错了
Skill 输入映射错了
Skill 输出结构不合理
Tool Result 回填有问题
最终答案阶段没有用好工具结果
```

如果没有 Run 详情页，我只能看最终答案。

但有了 Run 详情页，我能沿着事件顺序往回看。

看到模型什么时候开始分析。
看到它什么时候调用 Tool。
看到 Tool 参数是什么。
看到 Skill 输入是什么。
看到 Skill 输出是什么。
看到最终答案是怎么产生的。

这就有了排查依据。

---

这个页面其实是一个轻量版 Trace。

公司里可能会用 Coze Loop 或更完整的观测平台。

但我现在不想一上来就接那些平台化工具。

因为 MVP 阶段最重要的是先理解 Trace 的价值。

所以我先自己做一个最小版本：

```txt
Run 详情页
AgentEvent 列表
Tool / Skill 过程卡片
最终答案展示
```

它不复杂，但已经能让我解释清楚一次 Agent Run。

后面如果接 Coze Loop，我也知道哪些数据应该上报。

---

刚开始 Run 详情页读的是内存 run-store。

也就是说，只要服务没有重启，详情页就能看到这次 Run。

但这只是过渡方案。

因为内存数据有明显问题：

```txt
服务重启就没了
多实例之间不共享
不能长期复盘
不能做统计
不能做 replay
```

所以我后来又把 Run 详情页改成优先读数据库。

读数据库失败或者查不到时，再 fallback 到内存。

这个过渡方式比较稳。

既不影响当前 SSE 演示链路，又能逐步把数据持久化能力接进来。

---

这个阶段我也开始更重视 `traceId`。

`runId` 代表一次 Agent 执行。

`traceId` 更偏向排查链路。

在日志、数据库、前端页面里都带上它之后，我可以把一次执行过程串起来。

比如：

```txt
前端看到 runId
Run 详情页看到 traceId
CLS 日志里搜索 traceId
数据库里查 AgentEvent
```

这些信息就不再是分散的。

对一个 Agent 项目来说，可观察性不是上线之后才补的东西。

它应该从 MVP 阶段就开始留接口。

---

做完 Run 详情页后，我对“Agent Console”这个名字有了更明确的感觉。

它不是一个聊天页面。

它更像一个控制台。

用户在这里发起任务。
前端在这里观察任务。
开发者在这里复盘任务。
后续运维也可以从这里排查任务。

如果一个 Agent 系统没有这种复盘能力，那它出问题时会非常痛苦。

因为 AI 的不确定性本来就高。

工程侧能做的，就是尽量让每一步都留下痕迹。

Run 详情页就是这个痕迹的入口。
